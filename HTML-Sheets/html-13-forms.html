HTML · Sheet 13

Forms

form · input types · textarea · select · button · label · fieldset · validation

Description

Forms are the primary mechanism for collecting user input — from simple search boxes to multi-step checkouts. HTML5 provides over 20 input types, each with built-in validation, mobile keyboards, and accessibility semantics. Every input needs a <label> — either wrapping it or linked via for/id. Use <fieldset> + <legend> to group related controls. HTML's native validation (required, pattern, min/max) provides a solid baseline before adding JS validation.

Notes

  • method="GET" appends data to the URL — use for search; method="POST" sends in the body — use for sensitive data
  • Every form control must have an associated <label> — placeholder is not a substitute
  • autocomplete tokens (e.g. "email", "given-name") let password managers and browsers autofill correctly
  • <button type="button"> inside a form does NOT submit — omitting type defaults to submit
  • Use novalidate on <form> only when replacing browser validation with a custom JS implementation

Input Types Reference

type=Purpose & behaviour
textSingle-line free text — default type

passwordMasked text — never autocomplete="off" for passwords

dateDate picker (YYYY-MM-DD value)
timeTime picker (HH:MM value)
datetime-localDate + time, no timezone
monthMonth + year picker
rangeSlider; min, max, step, value
colorColor picker; value is hex
checkboxToggle; checked attribute = default on
radioOne-of-many; grouped by name attribute
fileFile picker; accept, multiple attributes
hiddenNot shown; submits value silently
submitSubmit button — prefer <button type="submit">
resetResets form to defaults — rarely useful

♿ Accessibility

  • Every input needs a visible <label> — placeholder disappears on typing and has low contrast; it is not a label
  • Group radio buttons and checkboxes in <fieldset> with <legend> — screen readers announce the legend before each option
  • Link error messages with aria-describedby="error-id" — screen readers read the error after the field's label when focused
  • Use aria-invalid="true" on fields with validation errors — announced by screen readers alongside the field role

Example

<form action="/signup" method="post">

  <!-- ① Text + email — label linked via for/id pair -->
  <label for="name">Full name</label>
  <input type="text"  id="name"  name="name"
         autocomplete="name" required />

  <label for="email">Email address</label>
  <input type="email" id="email" name="email"
         autocomplete="email" required />

  <!-- ② Password with accessible hint -->
  <label for="pwd">Password</label>
  <input type="password" id="pwd" name="password"
         minlength="8" autocomplete="new-password"
         aria-describedby="pwd-hint" required />
  <p id="pwd-hint">Minimum 8 characters.</p>

  <!-- ③ Select dropdown with optgroup -->
  <label for="country">Country</label>
  <select id="country" name="country" autocomplete="country-name">
    <option value="">Select a country</option>
    <optgroup label="North America">
      <option value="US">United States</option>
      <option value="CA">Canada</option>
    </optgroup>
  </select>

  <!-- ④ Textarea -->
  <label for="bio">Bio</label>
  <textarea id="bio" name="bio" rows="4" maxlength="500"
            placeholder="Tell us about yourself…"></textarea>

  <!-- ⑤ Radio group — fieldset + legend required for a11y -->
  <fieldset>
    <legend>Preferred contact method</legend>
    <label><input type="radio" name="contact" value="email" checked /> Email</label>
    <label><input type="radio" name="contact" value="phone" /> Phone</label>
  </fieldset>

  <!-- ⑥ Checkbox — wrapping label gives larger click target -->
  <label>
    <input type="checkbox" name="newsletter" value="yes" />
    Subscribe to newsletter
  </label>

  <!-- ⑦ File upload -->
  <label for="avatar">Profile photo</label>
  <input type="file" id="avatar" name="avatar"
         accept="image/png, image/jpeg, image/webp" />

  <!-- ⑧ Hidden field — submitted silently, not shown to user -->
  <input type="hidden" name="csrf_token" value="abc123xyz" />

  <!-- ⑨ Built-in validation attributes -->
  <input type="number" min="1" max="100" />
  <input type="text"   pattern="[A-Z]{3}-\d{4}"
         title="Format: ABC-1234" />

  <!-- ⑩ Buttons — always specify type inside a form -->
  <button type="button">Cancel</button>   <!-- does NOT submit -->
  <button type="submit">Create Account</button>

</form>

⚠ Common Mistakes

  • Using placeholder as a label — it disappears on typing and fails WCAG contrast requirements
  • Omitting type on <button> inside a form — defaults to submit, causing accidental submissions
  • Grouping radio buttons without <fieldset>/<legend> — screen readers read each option without the group question
  • Using autocomplete="off" on password fields — prevents password managers from filling saved credentials
  • Displaying validation errors without linking them to the field via aria-describedby — screen reader users miss the error message

✓ Tips & Best Practices

  • Always use the most specific input type — type="email" gives free format validation and the right mobile keyboard
  • Add autocomplete tokens to every input — enables autofill for users and is a WCAG 1.3.5 requirement
  • Use the wrapping label pattern for checkboxes and radios — larger click target, no for/id pairing needed
  • Test all forms with keyboard-only navigation — Tab between fields, Space/Enter for checkboxes and submit

Quick Summary

  • Every input needs a <label> — placeholder is not a substitute
  • Group radio/checkbox sets in <fieldset> + <legend>
  • Always specify type on <button> — default is submit
  • Use HTML5 input types for free validation, mobile keyboards, and semantic meaning
