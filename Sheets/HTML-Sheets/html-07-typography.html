HTML · Sheet 7

Typography

headings · paragraphs · emphasis · code · inline text elements

Description

HTML provides a rich set of text-level elements that give meaning to written content. The key distinction is semantic vs presentational: <strong> means "this is important" (screen readers stress it); <b> means "bold visually" (no semantic weight). Always choose the element that describes what the text means, not how you want it to look — CSS handles appearance. Heading levels (<h1>–<h6>) define the document outline used by search engines, assistive technology, and browser reading modes.

Notes

  • One <h1> per page — it is the document title; all other headings descend from it
  • Never skip heading levels for visual sizing — use CSS for that; skipping <h2> to <h4> breaks the outline
  • <strong> and <em> are semantic; <b> and <i> are presentational — prefer the semantic versions
  • <pre> preserves whitespace and line breaks exactly as written — used to wrap <code> blocks
  • <abbr title> exposes the full form on hover and to screen readers — use it on first occurrence

Inline Text Elements

Element Meaning Default render
<strong> Strong importance Bold
<em> Stress emphasis Italic
<b> Stylistically bold, no meaning Bold
<i> Offset text (term, thought) Italic
<mark> Highlighted / search result Yellow highlight
<del> Deleted text Strikethrough
<ins> Inserted text Underline
<sub> Subscript Below baseline, smaller
<sup> Superscript Above baseline, smaller
<code> Inline code fragment Monospace
<kbd> Keyboard input Monospace
<abbr> Abbreviation with expansion Dotted underline

♿ Accessibility

  • Screen readers announce <strong> with increased stress and <em> with emphasis — <b> and <i> are ignored
  • Heading levels define the document outline — screen reader users navigate pages by jumping between headings; a broken outline is a major barrier
  • <abbr title="..."> exposes the expansion to screen readers and on hover — use on first occurrence of every abbreviation
  • Never use headings purely for visual size — a user relying on heading navigation will be confused by out-of-order levels

Example

<!-- ① Heading hierarchy — one h1, then descend logically -->
<h1>The Guide to HTML Typography</h1>
  <h2>Text Elements</h2>
    <h3>Semantic vs Presentational</h3>
<!-- Never skip levels (h2 → h4) just for visual sizing — use CSS -->

<!-- ② Paragraph -->
<p><!-- default block of body text --></p>

<!-- ③ Semantic emphasis — read by screen readers with stress -->
<p>
  This step is <strong>required</strong> before continuing.
  Read <em>all</em> instructions carefully.
</p>

<!-- ④ Presentational — visual only, no screen reader stress -->
<p>
  The term <i>terroir</i> refers to environmental factors.
  Significant results shown in <b>bold</b>.
</p>

<!-- ⑤ mark: highlighted match (e.g. search result) -->
<p>Results for <mark>semantic HTML</mark> — 42 articles found.</p>

<!-- ⑥ del + ins: tracked changes, price updates -->
<p>Was <del>$49/mo</del> now <ins>$29/mo</ins></p>

<!-- ⑦ sub + sup -->
<p>Water is H<sub>2</sub>O.   E = mc<sup>2</sup></p>
<p>See footnote<sup><a href="#fn1">1</a></sup></p>

<!-- ⑧ code: inline code fragment -->
<p>Use <code>querySelector()</code> to find elements.</p>

<!-- ⑨ pre + code: multi-line block — whitespace preserved exactly -->
<pre><code>function greet(name) {
  return `Hello, ${name}!`;
}
greet('World'); // "Hello, World!"</code></pre>

<!-- ⑩ kbd: keyboard instruction -->
<p>Save with <kbd>Ctrl</kbd> + <kbd>S</kbd></p>

<!-- ⑪ abbr: expand abbreviation on first use -->
<p>
  <abbr title="HyperText Markup Language">HTML</abbr> is the
  language of the web.
</p>

⚠ Common Mistakes

  • Skipping heading levels for visual effect — jumping from <h2> to <h4> breaks the document outline
  • Using multiple <h1> elements — one per page is the standard; it signals the single main topic
  • Using <b> or <i> when <strong> or <em> is semantically correct
  • Using <code> without <pre> for multi-line blocks — whitespace collapses without <pre>
  • Using headings to style decorative text — use CSS classes instead

✓ Tips & Best Practices

  • Build your heading outline first, without any CSS — it should read like a table of contents
  • Use <strong> sparingly — if everything is important, nothing is
  • Use <abbr> on first occurrence only — repeating it on every instance is noisy
  • Pair <del> + <ins> with aria-label or screen-reader text since strikethrough is visual-only

Quick Summary

  • One <h1> per page; headings <h2>–<h6> must descend logically — never skip levels
  • <strong>/<em> = semantic weight; <b>/<i> = visual only
  • <pre><code> for multi-line code; <code> alone for inline fragments
  • Choose elements for meaning first — CSS will handle the look
