Recursive Functions

A recursive function calls itself with a smaller or simpler version of the problem until it reaches a base case that stops the recursion. Every recursive call adds a frame to the call stack — too many nested calls without bottoming out causes a RangeError: Maximum call stack size exceeded. Recursion is natural for problems with self-similar structure: tree/graph traversal, nested data, divide-and-conquer algorithms, and mathematical sequences. For flat repeated work, a loop is almost always more efficient. JavaScript does not perform automatic tail-call optimisation in most engines — deep recursion should be converted to iteration or use a trampoline for production use.

Recursion Checklist

  • Always define a base case that stops recursion
  • Each call must move closer to the base case
  • Call stack limit: ~10,000–15,000 frames in Node/V8
  • Deep recursion → convert to iteration or use a stack
  • Overlapping subproblems → add memoisation
  • Tree/graph traversal is recursion's strongest use case
  • Mutual recursion: A calls B, B calls A

Recursion Examples

// ── Classic: factorial ────────────────────────────────────────
function factorial(n) {
  if (n <= 1) return 1;          // base case
  return n * factorial(n - 1);  // recursive case
}
factorial(5);                    // 120

// ── Fibonacci (naive — exponential time) ─────────────────────
function fib(n) {
  if (n <= 1) return n;          // base cases: fib(0)=0, fib(1)=1
  return fib(n - 1) + fib(n - 2);
}
fib(10);                         // 55

// ── Fibonacci with memoisation (linear time) ─────────────────
function fibMemo(n, memo = {}) {
  if (n in memo) return memo[n];
  if (n <= 1) return n;
  memo[n] = fibMemo(n - 1, memo) + fibMemo(n - 2, memo);
  return memo[n];
}
fibMemo(40);                     // 102334155  (fast)

// ── Tree traversal ────────────────────────────────────────────
const tree = {
  value: 1,
  children: [
    { value: 2, children: [{ value: 4, children: [] }] },
    { value: 3, children: [{ value: 5, children: [] }] },
  ],
};

function sumTree(node) {
  return node.value + node.children.reduce(
    (sum, child) => sum + sumTree(child), 0
  );
}
sumTree(tree);                   // 15

// ── Flatten nested array ──────────────────────────────────────
function flatten(arr) {
  return arr.reduce((flat, item) =>
    Array.isArray(item)
      ? flat.concat(flatten(item))
      : flat.concat(item),
  []);
}
flatten([1, [2, [3, [4]]]]);    // [1, 2, 3, 4]
// Modern alternative: [1, [2, [3]]].flat(Infinity)

// ── Deep object clone (educational — use structuredClone) ─────
function deepClone(value) {
  if (Array.isArray(value)) return value.map(deepClone);
  if (value && typeof value === "object") {
    return Object.fromEntries(
      Object.entries(value).map(([k, v]) => [k, deepClone(v)])
    );
  }
  return value;
}

// ── Converting recursion to iteration (stack-based) ──────────
function sumTreeIter(root) {
  const stack = [root];
  let total = 0;
  while (stack.length) {
    const node = stack.pop();
    total += node.value;
    stack.push(...node.children);
  }
  return total;
}
sumTreeIter(tree);               // 15  — no call stack risk

Notes

Base case must be reachable — if the problem never reduces to the base case, you get infinite recursion and a stack overflow
Naive Fibonacci is O(2ⁿ) — each call spawns two more; memoisation reduces it to O(n) by caching already-computed values
Tree traversal is where recursion shines — the recursive structure of the code mirrors the recursive structure of the data
V8 (Node/Chrome) call stack limit is roughly 10,000–15,000 frames — deep recursion on large inputs needs iteration
Stack-based iteration — replace the call stack with an explicit array; gives you the same result without stack overflow risk
structuredClone(obj) is the modern built-in for deep cloning — no need to write your own for most cases

Common Mistakes

  • Missing or unreachable base case — causes infinite recursion and a stack overflow crash
  • Using naive recursion for Fibonacci or similar on large inputs — exponential time; add memoisation
  • Forgetting that JS lacks automatic tail-call optimisation in most engines — deep tail recursion still overflows
  • Writing recursive code when a simple loop would suffice — recursion adds overhead and risk for flat problems

Tips

  • Add memoisation to any recursive function with overlapping subproblems — it is a one-step transformation that can change O(2ⁿ) to O(n)
  • For production tree/graph traversal on large data, convert to an explicit stack — same logic, no stack overflow risk
  • Use arr.flat(Infinity) and structuredClone(obj) instead of hand-written recursive flatten/clone

Summary

  • Recursion requires a base case and progress toward it on every call
  • Best for: tree/graph traversal · nested data · divide-and-conquer
  • Add memoisation for overlapping subproblems · convert to iteration for deep recursion
  • V8 stack limit ~10k frames — large inputs need iterative solutions
