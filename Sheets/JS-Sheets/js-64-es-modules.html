ES Modules (ESM)

ES Modules are the official JavaScript module system, introduced in ES2015. Every ESM file has its own scope — nothing leaks to global unless explicitly exported. Named exports allow multiple exports per file; default exports provide a single primary export. Imports are static — resolved at parse time, which enables tree-shaking. Dynamic import() loads modules on demand at runtime, returning a Promise — essential for code-splitting and lazy loading. In browsers, activate with <script type="module">; in Node, use "type": "module" in package.json or .mjs extensions.

Key Points

  • ESM files always run in strict mode
  • Each file has its own scope — no globals leaked
  • Static imports are hoisted and resolved first
  • Named exports: multiple per file · Default: one per file
  • Node ESM requires .js extension in import paths
  • Dynamic import() returns a Promise
  • Live bindings — imported values reflect changes in the exporting module

ESM Examples

// ── Named exports (math.js) ───────────────────────────────────
export const PI = 3.14159;
export function add(a, b) { return a + b; }
export function sub(a, b) { return a - b; }
export class Vector { /* ... */ }

// Export list at bottom (also valid)
const TAU = PI * 2;
function mul(a, b) { return a * b; }
export { TAU, mul };

// ── Default export (logger.js) ────────────────────────────────
export default class Logger {
  log(msg) { console.log(msg); }
}
// One default per file — can be any expression

// ── Named imports ─────────────────────────────────────────────
import { PI, add, Vector } from "./math.js";    // exact names
import { add as sum } from "./math.js";          // rename
import * as math from "./math.js";               // namespace import
math.PI;
math.add(1, 2);

// ── Default import ────────────────────────────────────────────
import Logger from "./logger.js";               // any name works
import MyLog from "./logger.js";                // same default

// ── Mixed default + named ─────────────────────────────────────
// (utils.js): export default fn; export const VERSION = "1.0";
import fn, { VERSION } from "./utils.js";

// ── Re-exporting (barrel file: index.js) ─────────────────────
export { add, sub } from "./math.js";            // re-export named
export { default as Logger } from "./logger.js"; // re-export default as named
export * from "./helpers.js";                    // re-export everything

// ── Dynamic import() — runtime lazy loading ───────────────────
const btn = document.querySelector("#load");
btn.addEventListener("click", async () => {
  const { Chart } = await import("./chart.js"); // loaded on click
  new Chart(canvas).render(data);
});

// Conditional import
if (process.env.DEBUG) {
  const { debug } = await import("./debug-tools.js");
  debug.enable();
}

// ── TypeScript: import types ──────────────────────────────────
import type { User } from "./types.js";   // type-only; erased at compile
import { type Config, loadConfig } from "./config.js"; // mixed

// ── Node ESM: must include .js extension ─────────────────────
import { helper } from "./utils.js";      // ✔ explicit extension required
import { helper } from "./utils";         // ✖ Error in Node ESM

Notes

Static imports are hoisted — they are resolved before any code runs; you can't put them inside an if block
Live bindings — if a module changes an exported variable, all importers see the updated value; this differs from CommonJS which copies values
Circular dependencies — ESM handles them via live bindings; variables may be undefined at first if the circular dep hasn't run yet
Barrel files (index.js) re-export from multiple modules to provide a clean public API — can hurt tree-shaking if not careful
import type (TS) — completely erased at compile time; prevents accidentally importing a type as a value and causing runtime issues
Node ESM requires explicit .js extensions even for TypeScript files — TS resolves .ts files but the compiled output needs .js

Common Mistakes

  • Omitting the .js extension in Node ESM imports — Node doesn't auto-resolve bare specifiers in ESM
  • Using import inside a function expecting it to be synchronous — static imports must be top-level; use dynamic import() for conditional/lazy loading
  • Mixing default and named exports carelessly — a file can have both, but it can be confusing; pick one style per file
  • Circular imports causing undefined values — restructure to break the cycle or use dynamic imports

Tips

  • Prefer named exports over default — they are explicitly named at the import site, making code easier to search and refactor
  • Use dynamic import() for large optional features — splits the bundle so users don't download code they don't use
  • Use import type in TypeScript for all type-only imports — keeps the compiled output clean and prevents accidental value usage

Summary

  • Named: export const x · Default: export default x · Re-export: export { x } from
  • Static import — hoisted, tree-shakeable · Dynamic import() — lazy, returns Promise
  • ESM always strict · each file has its own scope · live bindings (not copies)
  • Node ESM requires .js extension · import type in TS erases at compile time
