Interface OOP Examples

// ── Interface as a contract ───────────────────────────────────
interface Drawable {
  draw(): void;
  readonly color: string;
}

interface Resizable {
  resize(factor: number): void;
}

// ── implements — class must satisfy the interface ─────────────
class Circle implements Drawable, Resizable {  // multiple interfaces
  constructor(
    public readonly color: string,
    private radius: number
  ) {}

  draw(): void {
    console.log(`Drawing ${this.color} circle r=${this.radius}`);
  }

  resize(factor: number): void {
    this.radius *= factor;
  }
}

// ── Structural typing — "duck typing" ─────────────────────────
// TS uses structural, not nominal, typing.
// Any object with the right shape satisfies an interface.
interface Printable {
  print(): string;
}

class Invoice {
  print() { return "Invoice #001"; }
  // Does NOT say `implements Printable` — but still satisfies it
}

function printAll(items: Printable[]): void {
  items.forEach(i => console.log(i.print()));
}

const inv = new Invoice();
printAll([inv]);               // ✔ structurally compatible

// Plain objects also satisfy interfaces:
const mockPrintable: Printable = { print: () => "Mock" };
printAll([mockPrintable]);     // ✔

// ── Interface with optional and readonly members ──────────────
interface Config {
  host: string;
  port: number;
  timeout?: number;            // optional
  readonly version: string;    // cannot be changed after set
}

// ── Extending interfaces ──────────────────────────────────────
interface Shape {
  area(): number;
}
interface ColoredShape extends Shape {
  color: string;
}
interface LabledShape extends ColoredShape {
  label: string;
}

class Triangle implements LabledShape {
  constructor(
    public color: string,
    public label: string,
    private base: number,
    private height: number
  ) {}
  area() { return 0.5 * this.base * this.height; }
}

// ── Using interfaces as function parameter types ──────────────
function renderShape(shape: Drawable & Resizable): void {
  shape.resize(2);
  shape.draw();
}
renderShape(new Circle("blue", 5));

// ── Interface for callbacks ───────────────────────────────────
interface EventHandler<T> {
  (event: T): void;
}
type ClickHandler = EventHandler<MouseEvent>;

Notes

Structural typing — TS checks shape, not class identity; an object satisfies an interface if it has the right properties and methods, regardless of how it was created
implements — optional but explicit; causes TS to error if the class fails to satisfy the interface, instead of silently missing a method
A class can implement multiple interfaces but extend only one class — interfaces are the mechanism for multiple-contract satisfaction
Interfaces are erased at compile time — they exist only as TS type information; no runtime footprint
Plain objects satisfy interfaces too — a common pattern for test mocks: const mock: MyInterface = { method: jest.fn() }
Intersection type Drawable & Resizable in a function parameter — requires the argument to satisfy both interfaces simultaneously

Common Mistakes

  • Using an abstract class when you only need a type contract — a plain interface is simpler and has no runtime cost
  • Forgetting that interfaces are structurally typed — a class with the right shape satisfies an interface even without implements
  • Confusing TS private in interface implementations with JS privacy — interface members are public; TS private only affects the class, not the interface contract

Tips

  • Use interfaces for capability contracts (Serializable, Comparable, Disposable) — they cleanly decouple implementation from behaviour
  • Use plain objects satisfying interfaces for test mocks — no need to create an actual class just for testing
  • Prefer interface over abstract class when you don't need shared implementation — it compiles away completely

Summary

  • implements — class satisfies interface; explicit but optional (structural typing applies anyway)
  • Structural typing — shape matters, not class identity; plain objects satisfy interfaces
  • Interfaces: multiple implements allowed · Abstract classes: single extends
  • Interfaces are erased at runtime — zero overhead; abstract classes exist at runtime
