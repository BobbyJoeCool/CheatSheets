Sheet 03

Program Structure

Entry points · script vs module mode · execution order · common patterns

How a JS/TS Program Starts

JavaScript has no required entry point function — execution starts at the top of whichever file the runtime loads first. In a browser that is the <script> tag in your HTML. In Node.js it is the file passed on the command line (node index.js). TypeScript is never executed directly — it must first be compiled by tsc, or run on-the-fly by tsx, ts-node, Deno, or Bun. Modern projects almost always use a bundler (Vite, esbuild) that resolves imports and outputs an optimised bundle for production.

Script vs Module Mode

  • Script mode — variables are global; no import/export; legacy default
  • Module mode — each file has its own scope; import/export available; always strict
  • Browser: type="module" on the script tag activates module mode
  • Node: "type":"module" in package.json makes all .js files ESM
  • Node: .mjs extension forces ESM; .cjs forces CommonJS
  • Modules are always in strict mode — no "use strict" needed

Structure Examples

// ── Minimal JS program (Node or browser) ────────────────────
// No boilerplate required — runs top to bottom
console.log("Hello, world!");              // Hello, world!

// ── Minimal TypeScript program ───────────────────────────────
// Identical to JS — type annotations are the only addition
const greeting: string = "Hello, TS!";
console.log(greeting);                     // Hello, TS!

// ── ES Module structure ──────────────────────────────────────
import { helper } from "./utils.js";       // named import
import defaultExport from "./lib.js";      // default import

export const version = "1.0.0";           // named export
export default function main() { }        // default export

// ── CommonJS structure (Node legacy) ────────────────────────
const fs = require("fs");                  // import
module.exports = { version };             // export object
exports.helper = function() { };          // shorthand export

// ── Common Node entry-point pattern ─────────────────────────
// Wrapping in async main() allows top-level await in older Node
async function main(): Promise<void> {
  console.log("App started");
}
main().catch(console.error);

// ── Modern: top-level await (ESM, Node 14.8+) ───────────────
// In .mjs or with "type":"module" — no wrapper needed
const data = await fetch("/api").then(r => r.json());
console.log(data);

Notes

JS executes top-to-bottom — there is no main() requirement
ESM import is async-capable and statically analysable (tree-shaking works)
CommonJS require() is synchronous — cannot be used inside async blocks
Cannot freely mix ESM and CJS in the same file — pick one per project
TS type annotations (: string, Promise<void>) are erased at compile time
Top-level await requires ESM mode — will error in CommonJS files

Common Mistakes

  • Forgetting type="module" in HTML — import fails with a syntax error
  • Missing "type":"module" in package.json when using ESM import in Node
  • Using require() inside an ESM file — will throw ReferenceError
  • Forgetting .js extension on relative imports in ESM Node — ./utils won't resolve

Tips

  • Use async function main() + main().catch(console.error) to handle top-level errors safely in older Node
  • Add "scripts": { "dev": "tsx src/index.ts" } to package.json for a one-command dev start
  • When in doubt, use ESM — it is the modern standard and works everywhere

Summary

  • JS executes top-to-bottom from the first loaded file — no entry point required
  • Module mode (import/export) is modern standard; always strict
  • CommonJS (require) is Node legacy — still widely used but being phased out
  • Top-level await works in ESM files (Node 14.8+, Deno, Bun)
