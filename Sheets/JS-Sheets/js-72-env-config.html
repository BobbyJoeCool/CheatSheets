Environment Variables & Config

Environment variables are the standard way to configure applications across environments without hardcoding values in source code. dotenv loads a .env file into process.env at startup — ideal for local development. In production, variables are injected by the hosting platform (Heroku, Vercel, AWS, Kubernetes). Never commit .env files to version control — use .env.example with placeholder values as documentation. TypeScript users should validate and type-cast process.env at startup using zod or a manual check, since all env values are string | undefined.

.env File Conventions

  • .env — local dev values; never commit
  • .env.example — placeholder template; commit this
  • .env.local — personal overrides; never commit
  • .env.production — prod defaults (no secrets)
  • .env.test — test environment values
  • All values are strings — convert numbers/booleans explicitly
  • Comments start with #
  • No spaces around =: KEY=value

Config Examples

// ── .env file ─────────────────────────────────────────────────
// DATABASE_URL=postgres://user:pass@localhost:5432/mydb
// PORT=3000
// API_KEY=secret-key-here
// DEBUG=true
// NODE_ENV=development

// ── dotenv — load .env into process.env ───────────────────────
// npm install dotenv
import "dotenv/config";              // auto-load .env (ESM)
// OR: import dotenv from "dotenv"; dotenv.config();

// Call before any other imports that need env vars
const port = Number(process.env.PORT ?? 3000);
const debug = process.env.DEBUG === "true"; // explicit boolean parse

// ── Node 20.6+ built-in .env loading ─────────────────────────
// node --env-file=.env src/index.js   (no dotenv needed!)

// ── Manual typed config object ────────────────────────────────
function getConfig() {
  const required = ["DATABASE_URL", "API_KEY"];
  for (const key of required) {
    if (!process.env[key]) {
      throw new Error(`Missing required env var: ${key}`);
    }
  }
  return {
    port:        Number(process.env.PORT ?? 3000),
    databaseUrl: process.env.DATABASE_URL,
    apiKey:      process.env.API_KEY,
    debug:       process.env.DEBUG === "true",
    nodeEnv:     process.env.NODE_ENV ?? "development",
  };
}
export const config = getConfig(); // run at startup; fails fast

// ── Zod-validated config (TypeScript) ────────────────────────
// npm install zod dotenv
import { z } from "zod";
import "dotenv/config";

const envSchema = z.object({
  PORT:         z.coerce.number().default(3000),  // coerce string→number
  DATABASE_URL: z.string().url(),
  API_KEY:      z.string().min(1),
  DEBUG:        z.coerce.boolean().default(false),
  NODE_ENV:     z.enum(["development","test","production"])
                 .default("development"),
});

const parsed = envSchema.safeParse(process.env);
if (!parsed.success) {
  console.error("Invalid env vars:", parsed.error.flatten().fieldErrors);
  process.exit(1);
}
export const env = parsed.data;
// env.PORT is number, env.DEBUG is boolean — fully typed!

// ── Accessing config throughout the app ──────────────────────
// Import the validated config object, never process.env directly
import { env } from "./config.js";
const server = app.listen(env.PORT);

Notes

All process.env values are string | undefined — always convert numbers with Number() and booleans with === "true"
Validate at startup with a fail-fast check — better to crash immediately with a clear error than to fail mysteriously at runtime when a config value is used
Zod z.coerce.number() — automatically converts the string "3000" to the number 3000; avoids manual conversion
Import a typed config object in application code — never access process.env.MY_VAR directly throughout the codebase; centralise it
dotenv/config (auto-load) must be imported before anything that reads process.env — put it first in your entry point
Node 20.6+ has built-in --env-file flag — no dotenv dependency needed for simple cases

Common Mistakes

  • Committing .env to version control — exposes secrets; add it to .gitignore immediately
  • Treating env vars as booleans directly: if (process.env.DEBUG) — an unset var is undefined (falsy) but the string "false" is truthy
  • Scattering process.env access throughout the codebase — centralise in one config file so validation happens once
  • Putting secrets in .env.example — it's committed; use placeholder values like API_KEY=your-key-here

Tips

  • Use zod to validate and type process.env at startup — one schema, full type safety, clear error messages
  • Add .env to .gitignore on day one — recovering from accidentally committed secrets is painful
  • Use a secret manager (AWS Secrets Manager, Vault, Doppler) in production — environment variables are fine for non-sensitive config, not for rotating secrets

Summary

  • dotenv loads .env into process.env; Node 20.6+ has --env-file built-in
  • All env values are strings — convert numbers and booleans explicitly
  • Validate at startup with zod or manual checks — fail fast on missing config
  • Never commit .env · commit .env.example with placeholder values
