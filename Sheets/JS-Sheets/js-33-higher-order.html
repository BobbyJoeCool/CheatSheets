Higher-Order Functions

A higher-order function (HOF) is any function that either accepts a function as an argument or returns a function as its result. Because JavaScript treats functions as first-class values — they can be stored in variables, passed as arguments, and returned from other functions — HOFs are central to the language. Common built-in HOFs include Array.map, filter, reduce, and sort. Custom HOFs are used for callbacks, factories, partial application, and function composition. These patterns let you write reusable, declarative code with less repetition.

Key Patterns

  • Callback — function passed as argument, called later
  • Factory — function that returns a configured function
  • Partial application — pre-fill some args, return rest
  • Currying — transform f(a,b) into f(a)(b)
  • Composition — chain functions: output of one is input of next
  • Decorator — wrap a function to add behaviour
  • All array HOFs: map · filter · reduce · sort · find · every · some

Higher-Order Function Examples

// ── Callback — function passed as argument ────────────────────
function repeat(n, action) {
  for (let i = 0; i < n; i++) action(i);
}
repeat(3, (i) => console.log(i)); // 0 1 2

// ── Factory — returns a configured function ───────────────────
function multiplier(factor) {
  return (n) => n * factor;
}
const double = multiplier(2);
const triple = multiplier(3);
double(5);                    // 10
triple(5);                    // 15

// ── Partial application — pre-fill arguments ──────────────────
function add(a, b) { return a + b; }
const add5 = add.bind(null, 5);  // pre-fill a = 5
add5(3);                         // 8
add5(10);                        // 15

// Manual partial application
function partial(fn, ...preArgs) {
  return (...laterArgs) => fn(...preArgs, ...laterArgs);
}
const add10 = partial(add, 10);
add10(7);                        // 17

// ── Currying — f(a,b) → f(a)(b) ──────────────────────────────
function curry(fn) {
  return function curried(...args) {
    if (args.length >= fn.length) return fn(...args);
    return (...more) => curried(...args, ...more);
  };
}
const curriedAdd = curry((a, b, c) => a + b + c);
curriedAdd(1)(2)(3);             // 6
curriedAdd(1, 2)(3);             // 6
curriedAdd(1)(2, 3);             // 6

// ── Function composition ──────────────────────────────────────
const compose = (...fns) => (x) => fns.reduceRight((v, f) => f(v), x);
const pipe    = (...fns) => (x) => fns.reduce((v, f) => f(v), x);

const trim    = (s) => s.trim();
const lower   = (s) => s.toLowerCase();
const slug    = (s) => s.replace(/\s+/g, "-");

const toSlug = pipe(trim, lower, slug);
toSlug("  Hello World  ");   // "hello-world"

// ── Decorator — wrap function to add behaviour ────────────────
function withLogging(fn) {
  return function(...args) {
    console.log(`Calling ${fn.name} with`, args);
    const result = fn(...args);
    console.log(`Result:`, result);
    return result;
  };
}
const loggedAdd = withLogging(add);
loggedAdd(2, 3);
// Calling add with [2, 3]
// Result: 5

// ── Memoisation ───────────────────────────────────────────────
function memoize(fn) {
  const cache = new Map();
  return function(...args) {
    const key = JSON.stringify(args);
    if (cache.has(key)) return cache.get(key);
    const result = fn(...args);
    cache.set(key, result);
    return result;
  };
}
const expensiveFn = memoize((n) => n * n);
expensiveFn(5);   // computed: 25
expensiveFn(5);   // cached:   25

Notes

pipe — left-to-right composition; compose — right-to-left; pipe is more intuitive for most uses
Partial application with bind(null, ...) — first arg is this; pass null when you don't need it
Currying enables point-free style and makes functions configurable without repeating arguments
The decorator pattern used here is distinct from TS/ES decorators (Sheet 37) — these are plain function wrappers
Memoisation — cache results by serialising args to a key; only safe for pure functions (same input always gives same output)
fn.length — the number of declared parameters; used by the curry helper to know when enough args have been provided

Common Mistakes

  • Memoising impure functions — if the function has side effects or depends on external state, cached results will be wrong
  • Over-currying simple code — currying adds mental overhead; use it where partial application genuinely helps reuse
  • Confusing compose (right-to-left) and pipe (left-to-right) — applying them in the wrong order gives unexpected results

Tips

  • Use pipe to build data transformation pipelines — each step is a focused single-purpose function
  • The decorator / wrapper pattern is a lightweight alternative to subclassing — add logging, timing, or retry logic without modifying the original function
  • Libraries like lodash/fp and ramda provide production-ready curry, compose, and partial utilities

Summary

  • HOFs accept or return functions — callbacks, factories, decorators, memoisation
  • Partial application — pre-fill args with bind or a closure
  • Currying — transform multi-arg function into chain of single-arg functions
  • pipe — compose left-to-right; output of each function feeds the next
