Cluster & PM2 Examples

// ── Node cluster module ───────────────────────────────────────
import cluster from "cluster";
import http    from "http";
import os      from "os";

const NUM_CPUS = os.cpus().length;

if (cluster.isPrimary) {
  console.log(`Primary ${process.pid} — forking ${NUM_CPUS} workers`);

  // Fork one worker per CPU
  for (let i = 0; i < NUM_CPUS; i++) {
    cluster.fork();
  }

  // Restart crashed workers
  cluster.on("exit", (worker, code, signal) => {
    console.warn(`Worker ${worker.process.pid} died (${signal || code}). Restarting...`);
    cluster.fork();
  });

} else {
  // Each worker runs this independently
  http.createServer((req, res) => {
    res.writeHead(200);
    res.end(`Worker ${process.pid} handled this\n`);
  }).listen(3000);

  console.log(`Worker ${process.pid} started`);
}

// ── PM2 ecosystem.config.js ───────────────────────────────────
// npm install -g pm2
module.exports = {
  apps: [
    {
      name:         "api",
      script:       "./dist/server.js",
      instances:    "max",             // one per CPU (cluster mode)
      exec_mode:    "cluster",
      watch:        false,             // don't watch in production
      max_memory_restart: "500M",      // restart if over 500MB RAM
      env: {
        NODE_ENV: "production",
        PORT:     3000,
      },
      env_development: {
        NODE_ENV: "development",
        PORT:     3001,
      },
      error_file:   "./logs/error.log",
      out_file:     "./logs/out.log",
      log_date_format: "YYYY-MM-DD HH:mm:ss",
    },
    {
      name:    "worker",
      script:  "./dist/queue-worker.js",
      instances: 2,
      exec_mode: "fork",              // not cluster — separate scripts
    }
  ]
};

// Start: pm2 start ecosystem.config.js --env production
// Zero-downtime deploy:
//   1. Build new code: npm run build
//   2. pm2 reload ecosystem.config.js --env production

// ── Graceful shutdown ─────────────────────────────────────────
// Handle SIGTERM from PM2 (sent before killing process)
process.on("SIGTERM", async () => {
  console.log("SIGTERM received — shutting down gracefully");
  server.close(async () => {          // stop accepting new requests
    await db.disconnect();            // close DB connections
    process.exit(0);
  });
  // Force exit after 10s if cleanup takes too long
  setTimeout(() => process.exit(1), 10_000);
});

Notes

Node cluster — the primary process distributes incoming connections across worker processes using round-robin (Linux) or OS-level load balancing (Windows)
Cluster vs Worker Threads — cluster forks separate processes (separate memory); workers share memory; cluster is for network servers, workers for CPU tasks
PM2 cluster mode — handles the clustering automatically; -i max or instances: "max" creates one process per logical CPU
pm2 reload — zero-downtime restart; workers are recycled one at a time so the app stays available during deployment
Graceful shutdown — PM2 sends SIGTERM before SIGKILL; handle it to finish in-flight requests before exiting
pm2 startup + pm2 save — generates and installs an OS init script so PM2 restarts your apps automatically after a server reboot

Common Mistakes

  • Using in-memory state (sessions, caches) in a clustered app — each worker has its own memory; use Redis for shared state
  • Forgetting pm2 save after adding processes — they won't restart after a reboot
  • Not handling SIGTERM — PM2 force-kills after timeout, dropping in-flight requests
  • Running more instances than CPU cores — context-switching overhead reduces performance

Tips

  • Use PM2 over manual cluster code — it handles restarts, logs, monitoring, and zero-downtime reloads out of the box
  • In containers (Docker/Kubernetes), run a single Node process per container — let the orchestrator handle scaling; PM2 cluster is for bare metal/VMs
  • Use pm2 monit for a live dashboard showing CPU, RAM, and restarts per process — great for quick diagnostics

Summary

  • Cluster — fork one worker per CPU; primary distributes connections; restart crashed workers
  • PM2 — production process manager; cluster mode, logs, monitoring, zero-downtime reload
  • pm2 start -i max · pm2 reload · pm2 startup && pm2 save
  • Handle SIGTERM for graceful shutdown · use Redis for shared state across workers
