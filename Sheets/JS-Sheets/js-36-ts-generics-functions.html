TS Generics in Functions

<T> syntax · inference · constraints · multiple params · generic return types

Generic Functions

Generics let you write functions that work with any type while still preserving type safety — no any needed. A generic type parameter (<T>) is declared in angle brackets before the parameter list and can be used in parameter types, the return type, and the function body. TypeScript infers the type argument in most cases — you rarely need to supply it explicitly. Constraints (extends) restrict what types T can be, giving you access to specific properties or methods. Multiple type parameters (<T, U>) handle functions that relate two different types.

Key Points

  • <T> declares a type parameter — like a variable for types
  • TS infers T from the argument — rarely need to specify
  • T extends SomeType — constrain what T can be
  • Multiple params: <T, U>, <K, V>
  • T extends keyof U — T must be a key of object U
  • Generic arrows in TSX files need a trailing comma: <T,>
  • Generics belong on functions that work with multiple types

Generic Function Examples

// ── Basic generic — identity function ────────────────────────
function identity<T>(value: T): T {
  return value;
}
identity(42);          // returns 42, type inferred as number
identity("hello");     // returns "hello", type inferred as string
identity<boolean>(true); // explicit type argument

// ── Generic array utilities ───────────────────────────────────
function first<T>(arr: T[]): T | undefined {
  return arr[0];
}
first([1, 2, 3]);      // number | undefined
first(["a", "b"]);     // string | undefined

function last<T>(arr: T[]): T | undefined {
  return arr[arr.length - 1];
}

function zip<T, U>(a: T[], b: U[]): [T, U][] {
  return a.map((val, i) => [val, b[i]]);
}
zip([1, 2], ["a", "b"]); // [[1, "a"], [2, "b"]]

// ── Constraints ───────────────────────────────────────────────
// T extends object — T must have at least these properties
function getLength<T extends { length: number }>(val: T): number {
  return val.length;               // ✔ T has .length
}
getLength("hello");   // 5
getLength([1, 2, 3]); // 3
getLength(42);        // ✖ number has no .length

// ── keyof constraint ─────────────────────────────────────────
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}
const user = { name: "Alice", age: 30 };
getProperty(user, "name");  // "Alice" — type is string
getProperty(user, "age");   // 30      — type is number
getProperty(user, "email"); // ✖ Error: not a key of user

// ── Generic with default type ─────────────────────────────────
function createPair<T = string>(a: T, b: T): [T, T] {
  return [a, b];
}
createPair("x", "y");   // ["x", "y"] — T inferred as string
createPair<number>(1, 2); // [1, 2]

// ── Generic return type inference ────────────────────────────
function merge<T extends object, U extends object>(a: T, b: U): T & U {
  return { ...a, ...b };
}
const result = merge({ name: "Alice" }, { age: 30 });
// result: { name: string } & { age: number }
result.name;  // ✔
result.age;   // ✔

// ── Generic callback type ────────────────────────────────────
function transform<T, U>(arr: T[], fn: (item: T) => U): U[] {
  return arr.map(fn);
}
transform([1, 2, 3], (n) => n.toString()); // ["1", "2", "3"]
transform(["a", "b"], (s) => s.length);    // [1, 1]

Notes

T is just a convention — any name works (Item, Key, Value); single letters are standard for simple cases
Type inference — TS deduces T from the argument type; explicit <Type> is only needed when inference fails or would be ambiguous
T extends { length: number } — structural constraint; any type with a length property satisfies it (string, array, typed array)
K extends keyof T — K must be one of the keys of T; the return type T[K] is then exactly the property's type
T & U intersection return — the merged object has all properties of both; TS tracks each field's type precisely
Generic callbacks fn: (item: T) => U let you write fully typed utility functions like map, filter, and reduce

Common Mistakes

  • Using any instead of a generic — loses all type information; generics preserve it
  • Over-constraining with extends — too tight a constraint rejects valid types; start loose and tighten as needed
  • Adding generics to functions that only ever work with one type — unnecessary complexity; use a concrete type instead
  • Writing <T> in a .tsx file without a trailing comma — parser reads it as JSX

Tips

  • Let TS infer the type argument — only pass <Type> explicitly when the compiler can't figure it out on its own
  • Use descriptive names for multiple type params — <Key, Value> is clearer than <T, U> when the roles are distinct
  • Use K extends keyof T for any function that accepts an object and one of its keys — it is type-safe and avoids string indexing errors

Summary

  • function fn<T>(arg: T): T — type parameter declared before params, used in signature
  • TS infers T from argument type — explicit annotation rarely needed
  • T extends Constraint — restrict what types are accepted
  • K extends keyof T — type-safe property access; return type is T[K]
