Variables

var · let · const · hoisting · temporal dead zone · block scope

Declaring Variables

JavaScript has three ways to declare variables: var, let, and const. var is the legacy keyword — function-scoped, hoisted to the top of its function, and allows re-declaration. let and const were introduced in ES6 and are block-scoped — they only exist inside the { } block they are declared in. const prevents reassignment of the binding itself but does not make objects or arrays immutable. In modern JS/TS, always prefer const by default and reach for let only when you need to reassign. Avoid var entirely.

var vs let vs const

  • var — function-scoped · hoisted · re-declarable · avoid
  • let — block-scoped · not hoisted (TDZ) · reassignable
  • const — block-scoped · not hoisted (TDZ) · not reassignable
  • const object properties can still be mutated
  • const array items can still be pushed/popped
  • Use Object.freeze() for true immutability
  • TS as const makes literals deeply readonly

Variable Examples

// ── const — use by default ───────────────────────────────────
const name = "Alice";
name = "Bob";                 // ✖ TypeError: assignment to constant
const PI = 3.14159;

// const object — binding is fixed, properties are not
const user = { name: "Alice", age: 30 };
user.age = 31;                // ✔ property mutation is allowed
user = {};                    // ✖ TypeError: cannot reassign const

// const array — binding is fixed, contents are not
const colors = ["red", "green"];
colors.push("blue");          // ✔ ["red", "green", "blue"]
colors = [];                  // ✖ TypeError

// ── let — use when reassignment is needed ────────────────────
let count = 0;
count = count + 1;            // ✔ reassignment allowed
count++;                      // ✔ same

let score;                    // ✔ declared but undefined
score = 10;                   // ✔ assigned later

// ── var — legacy, avoid in modern code ──────────────────────
var x = 1;
var x = 2;                    // ✔ re-declaration allowed (confusing)
console.log(x);               // 2

// var leaks out of blocks (not block-scoped)
if (true) { var leaked = "oops"; }
console.log(leaked);          // "oops" — still accessible!

// let/const are block-scoped
if (true) { let safe = "ok"; }
console.log(safe);            // ✖ ReferenceError

// ── Hoisting ─────────────────────────────────────────────────
console.log(hoisted);         // undefined (var is hoisted)
var hoisted = "value";

console.log(notHoisted);      // ✖ ReferenceError (TDZ)
let notHoisted = "value";

// ── TypeScript: type annotations ─────────────────────────────
const age: number = 30;
let username: string = "Alice";
let isActive: boolean = true;
let data: string | null = null;  // union type

// as const — makes literal types narrow and readonly
const direction = "north" as const; // type is "north", not string
const config = { port: 3000, host: "localhost" } as const;
// config.port is readonly — type is 3000, not number

Notes

Hoisting — var declarations are moved to the top of their function; the value is undefined until the line runs
Temporal Dead Zone (TDZ) — let/const exist in scope from the start of the block but cannot be accessed before their declaration line
const prevents rebinding — the variable always points to the same object/array, but that object's contents can change
Object.freeze(obj) — makes top-level properties non-writable; not deep (nested objects still mutable)
as const in TS — infers the narrowest possible type and marks everything readonly
TS will error if you try to assign a value that doesn't match the declared type

Common Mistakes

  • Using var in modern code — it has confusing scoping and hoisting behaviour
  • Expecting const to freeze an object — it only prevents reassignment of the binding
  • Accessing a let variable before its declaration — throws a TDZ ReferenceError
  • Declaring variables inside loops with var — all iterations share the same variable

Tips

  • Default to const — switch to let only when the linter tells you the value is never reassigned
  • ESLint rules no-var and prefer-const enforce good habits automatically
  • Use as const in TS for config objects and lookup tables to get literal types instead of wide primitives

Summary

  • const — block-scoped · no reassignment · use by default
  • let — block-scoped · reassignable · use when needed
  • var — function-scoped · hoisted · avoid in modern code
  • TDZ: let/const cannot be read before their declaration line
