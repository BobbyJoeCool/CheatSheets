Logging

Log levels · structured logging · pino · winston · child loggers · production patterns

Log Levels (low → high severity)

LevelNumericUse For
trace10Very detailed debug — function entry/exit, loop steps
debug20Debug info — variable values, logic branches
info30Normal operation — server started, request received
warn40Unexpected but recoverable — deprecated API, retrying
error50Operation failed — caught exception, 5xx response
fatal60Process is about to crash — unrecoverable error

Logging Examples

// ── pino — fast structured logger ────────────────────────────
// npm install pino pino-pretty
import pino from "pino";

const logger = pino({
  level: process.env.LOG_LEVEL ?? "info",
  transport: process.env.NODE_ENV !== "production"
    ? { target: "pino-pretty" }   // human-readable in dev
    : undefined,                  // raw JSON in production
});

logger.info("Server started");
// {"level":30,"time":1705312200000,"msg":"Server started"}

logger.info({ userId: 42, action: "login" }, "User logged in");
// {"level":30,"userId":42,"action":"login","msg":"User logged in"}

logger.warn({ retries: 3 }, "DB connection slow, retrying");
logger.error({ err, requestId }, "Failed to process request");

// ── Child logger — add context to all subsequent logs ─────────
const reqLogger = logger.child({ requestId: "abc-123", userId: 42 });
reqLogger.info("Processing order");
// {"requestId":"abc-123","userId":42,"msg":"Processing order"}
reqLogger.error({ err }, "Order failed");
// requestId and userId automatically included

// ── winston ───────────────────────────────────────────────────
// npm install winston
import winston from "winston";

const wlog = winston.createLogger({
  level: "info",
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json()
  ),
  transports: [
    new winston.transports.Console({
      format: winston.format.simple(),    // human-readable console
    }),
    new winston.transports.File({
      filename: "logs/error.log",
      level: "error",                     // only errors go here
    }),
    new winston.transports.File({
      filename: "logs/combined.log",       // all levels here
    }),
  ],
});

wlog.info("Server started", { port: 3000 });
wlog.error("Unhandled exception", { err: error.message });

// ── Structured logging best practices ─────────────────────────
// ✔ always include context as object fields
logger.info({ userId, orderId, amount }, "Payment processed");

// ✖ avoid string interpolation — can't be queried in log systems
logger.info(`Payment ${orderId} processed for user ${userId}`);

// ── Express/Fastify middleware ─────────────────────────────────
app.use((req, res, next) => {
  req.log = logger.child({ requestId: crypto.randomUUID() });
  req.log.info({ method: req.method, url: req.url }, "Incoming request");
  next();
});

Notes

Structured logging (JSON) — machine-readable; fields can be queried/filtered by log aggregators (Datadog, Splunk, Loki)
Child loggers — inherit the parent's config and add fixed context fields to every log line; use one per request for request-scoped context
pino-pretty — transforms JSON logs to human-readable coloured output; only use in development, never in production
Log level via environment variable — LOG_LEVEL=debug enables verbose logging in staging without code changes
Never log sensitive data — passwords, tokens, credit card numbers, PII; use allow-lists of safe fields rather than blocking specific ones
Always log errors with the full error object, not just error.message — stack traces are essential for debugging production issues

Common Mistakes

  • Using console.log in production — no log levels, no structured output, no transport control
  • String-interpolating log messages — "User ${id} failed" can't be aggregated or queried; use object fields instead
  • Using pino-pretty in production — adds significant overhead; use raw JSON and pretty-print in your terminal with | pino-pretty
  • Logging sensitive data — passwords, tokens, or PII in logs creates security and compliance issues

Tips

  • Create a request-scoped child logger in middleware — every log line for a request automatically includes requestId for correlation
  • Set level: "warn" in production — info-level logs are high-volume; only emit them in staging for debugging
  • Use pino for high-throughput APIs — it is ~10× faster than winston by avoiding synchronous serialisation

Summary

  • Levels: trace → debug → info → warn → error → fatal
  • pino — fastest, JSON-first, great for APIs · winston — flexible multi-transport
  • Structured logging — pass context as an object, not a string · queryable in log aggregators
  • Child loggers — add request/user context to all logs automatically
