Sheet 05

tsconfig.json

Compiler options · strict checks · output control · project setup

What is tsconfig.json?

tsconfig.json is the configuration file for the TypeScript compiler (tsc). It lives in the root of your project and controls what files are compiled, what JS version is emitted, how strict the type checking is, and where output goes. Running tsc --init generates a starter file with every option documented inline. When using Vite, esbuild, or Bun, tsc is often used for type-checking only (tsc --noEmit) while the bundler handles transpilation and output. You can extend a base config with "extends": "./tsconfig.base.json".

strict: true Enables…

  • noImplicitAny — error on implicit any type
  • strictNullChecks — null/undefined must be handled explicitly
  • strictFunctionTypes — stricter function param checking
  • strictBindCallApply — type-safe call/apply/bind
  • strictPropertyInitialization — class props must be set in constructor
  • noImplicitThis — error on this with implicit any
  • useUnknownInCatchVariables — catch vars are unknown not any

Annotated tsconfig.json

{
  "compilerOptions": {

    // ── Output target ────────────────────────────────────────
    "target": "ES2022",           // JS version to emit (ES5 → ESNext)
    "module": "ESNext",           // module system: CommonJS | ESNext | NodeNext
    "moduleResolution": "bundler",// how imports are resolved: node | bundler | nodenext
    "lib": ["ES2022", "DOM"],     // built-in type definitions to include

    // ── Strictness ───────────────────────────────────────────
    "strict": true,               // ✔ enable ALL strict checks (recommended)
    "noUncheckedIndexedAccess": true, // arr[0] is T | undefined (extra safe)
    "exactOptionalPropertyTypes": true, // opt props cannot be explicitly undefined

    // ── Emit ─────────────────────────────────────────────────
    "outDir": "./dist",           // compiled JS output directory
    "rootDir": "./src",           // source files root
    "sourceMap": true,            // generate .map files for debugging
    "declaration": true,          // emit .d.ts type declaration files
    "noEmit": false,              // set true to type-check only (CI use)

    // ── Interop ──────────────────────────────────────────────
    "esModuleInterop": true,      // allows default import of CJS modules
    "allowSyntheticDefaultImports": true, // implied by esModuleInterop
    "skipLibCheck": true,         // skip checking .d.ts files (faster builds)
    "resolveJsonModule": true,    // allows: import data from "./data.json"

    // ── Project ──────────────────────────────────────────────
    "baseUrl": ".",               // base for non-relative imports
    "paths": {                    // import aliases
      "@/*": ["./src/*"]
    }
  },

  "include": ["src/**/*"],        // files to compile
  "exclude": ["node_modules", "dist"],
  "extends": "./tsconfig.base.json" // inherit from another config
}

Key Options

target — controls what JS syntax is emitted, not what you can write
module: NodeNext — use for Node.js projects; respects .mjs/.cjs
moduleResolution: bundler — use with Vite/esbuild; allows extensionless imports
lib: ["DOM"] — required if you use browser APIs; omit for Node-only projects
sourceMap — essential for debugging TS in browser DevTools or Node
paths — enables import x from "@/utils" instead of "../../utils"
skipLibCheck — speeds up compilation; safe to enable in most projects

Common Mistakes

  • Omitting strict: true — you lose most of TypeScript's safety benefits
  • Setting target: "ES5" when your runtime supports ES2020+ — unnecessarily large output
  • Forgetting lib: ["DOM"] in a browser project — document and fetch won't type-check
  • Using paths aliases without configuring the bundler too — tsc resolves them, bundlers don't

Tips

  • Run tsc --init to generate a commented starter file — then trim what you don't need
  • Use tsc --noEmit in CI for type-checking without producing output files
  • Create a shared tsconfig.base.json and extends it in each package of a monorepo

Summary

  • tsconfig.json controls what tsc compiles and how strict it is
  • Always enable strict: true — one flag turns on ~7 essential checks
  • target sets JS output version; lib sets available type definitions
  • Use noEmit: true when a bundler handles transpilation and tsc is type-check only
