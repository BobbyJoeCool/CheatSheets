TS Type Assertions & Narrowing

as · satisfies · typeof · instanceof · in · type guards · discriminated unions

Assertions vs Narrowing

Type assertions (as SomeType) tell the TypeScript compiler to treat a value as a specific type — you are overriding its inference. They are a promise to the compiler, not a runtime conversion, and can be unsafe if wrong. Type narrowing is the opposite approach: you write runtime checks (typeof, instanceof, in) that TypeScript understands and uses to automatically refine the type inside that branch. Narrowing is always safer than assertion — prefer it whenever possible.

Narrowing Techniques

  • typeof x === "string" — primitive type check
  • x instanceof Date — class/constructor check
  • "prop" in obj — property existence check
  • Array.isArray(x) — array check
  • x != null — null + undefined guard
  • Discriminated union — check a shared literal field
  • Type guard function — x is SomeType return type

Assertions & Narrowing Examples

// ── Type assertions ──────────────────────────────────────────
const input = document.getElementById("name") as HTMLInputElement;
input.value;                   // ✔ TS knows it's an input element

const len = (someValue as string).length;

// Double assertion (escape hatch — use sparingly)
const x = someValue as unknown as SpecificType;

// Non-null assertion operator (!)
const el = document.getElementById("app")!; // assert non-null
el.innerHTML = "Hello";        // ✔ no null check needed

// ── satisfies — validate without widening ────────────────────
const config = {
  port: 3000,
  host: "localhost",
} satisfies Record<string, string | number>;
// config.port is still type 3000, not number | string
config.port.toFixed();         // ✔ TS knows it's a number

// ── typeof narrowing ─────────────────────────────────────────
function format(val: string | number): string {
  if (typeof val === "string") {
    return val.toUpperCase();  // ✔ val is string here
  }
  return val.toFixed(2);       // ✔ val is number here
}

// ── instanceof narrowing ─────────────────────────────────────
function log(err: unknown) {
  if (err instanceof Error) {
    console.error(err.message); // ✔ err is Error
  }
}

// ── "in" narrowing ───────────────────────────────────────────
type Dog = { bark: () => void };
type Cat = { meow: () => void };
function speak(animal: Dog | Cat) {
  if ("bark" in animal) {
    animal.bark();             // ✔ narrowed to Dog
  } else {
    animal.meow();             // ✔ narrowed to Cat
  }
}

// ── Discriminated union narrowing ────────────────────────────
type Circle = { kind: "circle"; radius: number };
type Square = { kind: "square"; side: number };
type Shape  = Circle | Square;

function area(s: Shape): number {
  switch (s.kind) {
    case "circle": return Math.PI * s.radius ** 2;
    case "square": return s.side ** 2;
  }
}

// ── Custom type guard ─────────────────────────────────────────
function isString(val: unknown): val is string {
  return typeof val === "string";
}
if (isString(maybeStr)) {
  maybeStr.toUpperCase();      // ✔ narrowed to string
}

Notes

as Type — compile-time only; no runtime conversion; wrong assertions cause silent runtime errors
! non-null assertion — tells TS the value is never null/undefined; throws if wrong
satisfies — validates a value matches a type without widening the inferred type (TS 4.9+)
Discriminated unions — give each variant a shared literal field (kind, type, tag) for clean narrowing
Type guard functions return val is Type — lets you encapsulate and reuse narrowing logic
as unknown as T — double assertion; bypasses TS completely; only for truly unavoidable cases

Common Mistakes

  • Overusing as to silence errors — it hides bugs; narrow with runtime checks instead
  • Using ! on values that actually can be null — causes runtime crashes
  • Forgetting that as is compile-time only — it doesn't convert the value at runtime
  • Using as any as a quick fix — defeats the purpose of TypeScript entirely

Tips

  • Prefer narrowing over assertion — it is safe at runtime, not just at compile time
  • Use satisfies instead of as for config objects — you get validation without losing precise types
  • Design APIs with discriminated unions — the kind/type field makes narrowing automatic and exhaustive

Summary

  • as Type — assertion; compile-time only; overrides inference; can be unsafe
  • x! — non-null assertion; use sparingly
  • satisfies — validates without widening; prefer over as for objects
  • Narrowing — typeof · instanceof · in · Array.isArray · discriminated unions · type guards
