Decorators (TS)

Class · method · property · parameter decorators · decorator factories · TC39 stage

TypeScript Decorators

Decorators are a stage 3 TC39 proposal (now in the ECMAScript standard) that TypeScript has supported experimentally for years. They are special functions prefixed with @ that attach metadata or modify the behaviour of classes, methods, properties, and parameters at definition time. There are two decorator systems in TypeScript: the legacy experimental system (enabled via experimentalDecorators: true in tsconfig — used by NestJS, Angular, TypeORM) and the new TC39/ES2023 decorators (enabled by default in TS 5.0+ without the flag). This sheet covers the modern TC39 style with notes on the legacy API where it differs.

Decorator Types

  • Class — decorates the whole class
  • Method — decorates a class method
  • Field — decorates a class property/field
  • Getter/Setter — decorates an accessor
  • Accessor — @accessor auto-generates get/set
  • Decorator factory — function that returns a decorator
  • Applied bottom-up when multiple decorators stack

Decorator Examples

// ── Class decorator ───────────────────────────────────────────
function sealed(constructor: Function) {
  Object.seal(constructor);
  Object.seal(constructor.prototype);
}

@sealed
class BankAccount {
  balance = 0;
}

// ── Method decorator — log execution time ────────────────────
function logTime(
  _target: any,
  name: string,
  descriptor: PropertyDescriptor
) {
  const original = descriptor.value;
  descriptor.value = function (...args: any[]) {
    const start = performance.now();
    const result = original.apply(this, args);
    console.log(`${name} took ${(performance.now() - start).toFixed(2)}ms`);
    return result;
  };
  return descriptor;
}

class DataService {
  @logTime
  processData(items: string[]) {
    return items.map((s) => s.toUpperCase());
  }
}

// ── Decorator factory — configurable ─────────────────────────
function retry(times: number) {
  return function (
    _target: any,
    _name: string,
    descriptor: PropertyDescriptor
  ) {
    const original = descriptor.value;
    descriptor.value = async function (...args: any[]) {
      for (let i = 0; i < times; i++) {
        try {
          return await original.apply(this, args);
        } catch (err) {
          if (i === times - 1) throw err;
          console.warn(`Retry ${i + 1}/${times}`);
        }
      }
    };
    return descriptor;
  };
}

class ApiClient {
  @retry(3)
  async fetchUser(id: string) {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return res.json();
  }
}

// ── Common framework decorators (NestJS/Angular style) ────────
// These use experimentalDecorators: true
@Controller("/users")           // route prefix
class UserController {
  @Get("/:id")                  // HTTP GET handler
  @UseGuards(AuthGuard)         // apply middleware/guard
  async getUser(@Param("id") id: string) {
    return this.userService.find(id);
  }
}

// ── Property/field decorator ──────────────────────────────────
function readonly(
  _target: any,
  propertyKey: string
) {
  Object.defineProperty(_target, propertyKey, { writable: false });
}

class Config {
  @readonly
  apiVersion = "v2";
}

Notes

Decorators run at class definition time, not when instances are created — they modify the class/method in place
Method decorators receive (target, propertyKey, descriptor) — modify descriptor.value to wrap the method
Decorator factories — a function that takes config params and returns the actual decorator function; enables @retry(3) syntax
Multiple decorators stack bottom-up: @A @B method applies B first, then A wraps the result
Legacy (experimentalDecorators) vs TC39 decorators have slightly different signatures — check which your framework uses
NestJS, Angular, TypeORM, and MikroORM all rely heavily on decorators — in those ecosystems, experimentalDecorators: true is standard

Common Mistakes

  • Using legacy decorator syntax with TS 5+ without experimentalDecorators: true — different APIs, will break
  • Forgetting that decorators run at definition time — side effects happen once when the module loads, not per instance
  • Mutating descriptor.value without returning the descriptor — the modification won't apply
  • Using decorators in plain .js files — decorators require TypeScript or Babel with the decorator transform

Tips

  • Use decorator factories (@decorator(config)) rather than plain decorators — they are more flexible and reusable
  • Keep decorators focused — one concern per decorator (logging, caching, validation, auth) makes them composable
  • Check your framework docs for which decorator system it expects — NestJS and Angular pin to experimentalDecorators

Summary

  • @decorator — runs at class definition time; modifies class, method, field, or accessor
  • Method decorators — receive (target, name, descriptor); wrap descriptor.value
  • Decorator factories — @factory(config) returns the real decorator with config baked in
  • Legacy (experimentalDecorators) used by NestJS/Angular · TC39 style is the modern standard (TS 5+)
