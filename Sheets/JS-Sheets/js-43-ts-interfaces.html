Interface & Type Examples

// ── interface — object shape ──────────────────────────────────
interface User {
  id: number;
  name: string;
  email?: string;           // optional — string | undefined
  readonly createdAt: Date; // cannot be reassigned after creation
}

const alice: User = { id: 1, name: "Alice", createdAt: new Date() };
alice.createdAt = new Date(); // ✖ Error: readonly

// ── type alias ────────────────────────────────────────────────
type ID = string | number;  // union — only possible with type
type Point = { x: number; y: number };
type Nullable<T> = T | null;

// ── Extending interfaces ──────────────────────────────────────
interface Animal {
  name: string;
}
interface Dog extends Animal {
  breed: string;
}
const d: Dog = { name: "Rex", breed: "Lab" };

// ── Intersection with type ────────────────────────────────────
type Named  = { name: string };
type Aged   = { age: number };
type Person = Named & Aged;    // must have both

// ── Declaration merging (interface only) ─────────────────────
interface Window {
  myPlugin: () => void;        // add to existing Window type
}

// ── Index signature — dynamic keys ───────────────────────────
interface StringMap {
  [key: string]: string;       // any string key → string value
}
const headers: StringMap = { "Content-Type": "application/json" };

// Mixing known + dynamic keys
interface Config {
  name: string;                // known key
  [key: string]: unknown;      // dynamic keys — must be ≥ known types
}

// ── Record<K, V> — shorthand for index signature ─────────────
type Scores  = Record<string, number>;
type Status  = Record<"active" | "inactive" | "pending", boolean>;

const sc: Scores = { alice: 95, bob: 87 };
const st: Status = { active: true, inactive: false, pending: false };

// ── Readonly object type ──────────────────────────────────────
interface ImmutablePoint {
  readonly x: number;
  readonly y: number;
}
// Or with utility type:
type FrozenUser = Readonly<User>;

// ── Function signature in interface ──────────────────────────
interface Formatter {
  format(value: string): string;
  (value: string): string;     // callable signature
}

Notes

Convention: use interface for object shapes (especially public APIs); use type for unions, aliases, and computed types
Declaration merging — defining an interface with the same name twice merges the properties; type aliases cannot be redeclared
readonly in interfaces/types is compile-time only — the underlying JS object can still be mutated if you bypass TS
Index signature [key: string]: V — all known properties must be assignable to V; use unknown when values vary
Record<K, V> is shorthand for an index signature with a specific key type — cleaner than writing the signature manually
Both interface and type can be used with implements in a class — pick whichever is cleaner for the use case

Common Mistakes

  • Using interface for union types — interfaces cannot express A | B; use type
  • Adding an index signature with string value when known properties are number — TS requires all properties match the index type
  • Assuming readonly provides runtime immutability — it is a compile-time constraint only
  • Accidentally relying on declaration merging when you want a fresh type — define a new type or use a different name

Tips

  • Use Record<string, unknown> for generic objects of unknown shape — safer than { [key: string]: any }
  • Prefer interface for library public types — declaration merging lets consumers augment your types
  • Use Readonly<T> utility type on function parameters to signal that the function won't mutate its input

Summary

  • interface — object shapes, extends, declaration merging; preferred for public APIs
  • type — unions, intersections, aliases, computed types; more flexible
  • optional? · readonly · index signatures · Record<K,V>
  • Both work with implements in classes; both support extends / & composition
