Custom Errors & TypeScript

Custom error classes · error chaining · TS typed catch · cause · Result pattern

Custom Error Classes

Extending the built-in Error class lets you create domain-specific error types that can be distinguished with instanceof and carry additional context. Custom errors are invaluable in larger codebases — a ValidationError tells you immediately what went wrong without reading the message. ES2022 introduced error chaining via the cause option, allowing you to wrap a low-level error with higher-level context while preserving the original. In TypeScript, catch variables are typed unknown (with strict: true) — you must narrow the type before accessing .message or other properties.

Key Points

  • Extend Error to create domain-specific error types
  • Set this.name in constructor for correct err.name
  • Error({ cause }) — chains errors (ES2022+)
  • TS strict: catch var is unknown, not any
  • Narrow with instanceof Error before accessing .message
  • Result pattern — return { ok, value, error } instead of throwing
  • Custom errors are distinguishable with instanceof

Custom Error & TS Examples

// ── Custom error class ────────────────────────────────────────
class ValidationError extends Error {
  constructor(
    message: string,
    public readonly field: string  // extra context
  ) {
    super(message);
    this.name = "ValidationError"; // important: set name
  }
}

class NotFoundError extends Error {
  constructor(public readonly resource: string, id: string | number) {
    super(`${resource} with id "${id}" not found`);
    this.name = "NotFoundError";
  }
}

class NetworkError extends Error {
  constructor(message: string, public readonly statusCode: number) {
    super(message);
    this.name = "NetworkError";
  }
}

// Using custom errors
function validateAge(age: number) {
  if (age < 0 || age > 150) {
    throw new ValidationError("Age out of valid range", "age");
  }
}

try {
  validateAge(-5);
} catch (err) {
  if (err instanceof ValidationError) {
    console.error(`Invalid field "${err.field}":`, err.message);
  } else {
    throw err;
  }
}

// ── Error chaining with cause (ES2022) ────────────────────────
async function fetchUser(id: string) {
  try {
    const res = await fetch(`/api/users/${id}`);
    return await res.json();
  } catch (err) {
    throw new NetworkError("Failed to fetch user", 503, { cause: err });
    // err.cause preserves the original error
  }
}

// Access the cause:
try {
  await fetchUser("123");
} catch (err) {
  if (err instanceof NetworkError) {
    console.error("Status:", err.statusCode);
    console.error("Caused by:", (err as any).cause);
  }
}

// ── TypeScript: typing catch blocks ──────────────────────────
// With strict: true, caught errors are `unknown`, not `any`
try {
  riskyOp();
} catch (err: unknown) {
  if (err instanceof Error) {
    console.error(err.message);    // ✔ safe
  } else {
    console.error("Unknown error:", String(err));
  }
}

// ── Result pattern — avoid throwing for expected failures ─────
type Result<T> =
  | { ok: true;  value: T }
  | { ok: false; error: string };

function parseNumber(input: string): Result<number> {
  const n = Number(input);
  if (isNaN(n)) return { ok: false, error: `"${input}" is not a number` };
  return { ok: true, value: n };
}

const result = parseNumber("abc");
if (!result.ok) {
  console.error(result.error);     // "abc" is not a number
} else {
  console.log(result.value * 2);
}

Notes

Always set this.name = "ClassName" in the constructor — otherwise err.name shows "Error" instead of your class name
cause option — pass { cause: originalError } to preserve the full error chain for debugging
TS strict catch is unknown — you cannot call err.message without narrowing first; use instanceof Error
Custom errors work with instanceof across the whole codebase — much better than matching on err.message strings
Result pattern — avoids exceptions for expected failures (validation, parsing); exceptions are for unexpected failures
In Node.js 16+, error.cause is part of the spec; polyfill needed for older environments

Common Mistakes

  • Forgetting this.name = "ClassName" — custom error shows as "Error" in logs
  • Accessing err.message in TS without narrowing — unknown type has no properties
  • Throwing custom errors without extending Error — breaks instanceof checks and stack traces
  • Using exceptions for expected validation failures — use the Result pattern instead to keep control flow explicit

Tips

  • Create an error hierarchy: AppError → ValidationError, NotFoundError, NetworkError — catch AppError at the top level
  • Use the Result pattern for functions that can predictably fail — parseX(), validateX(), lookupX()
  • Log err.cause in your top-level error handler to always see the full chain of what went wrong

Summary

  • Extend Error for custom types · always set this.name · add extra fields as needed
  • new Error("msg", { cause: err }) — chains errors for full context (ES2022)
  • TS strict catch: unknown — narrow with instanceof Error before accessing properties
  • Result pattern — return { ok, value/error } for predictable failures; throw for unexpected ones
