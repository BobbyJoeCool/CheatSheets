Advanced API Patterns

Production API clients need more than basic fetch calls. Authentication headers must be attached to every protected request — an axios interceptor or a fetch wrapper handles this centrally. Rate limiting on the client side prevents hammering APIs and getting blocked. Retry with exponential backoff handles transient failures gracefully. AbortController signals prevent stale responses updating state after navigation. On the server side, CORS headers control which origins can call your API and must be configured correctly for browser clients.

Auth Patterns

  • Bearer token — Authorization: Bearer <token>
  • API key header — X-API-Key: <key>
  • Basic auth — Authorization: Basic <base64>
  • Cookie — httpOnly; sent automatically by browser
  • Store tokens in memory or httpOnly cookies — never localStorage for sensitive tokens
  • Refresh tokens in a 401 interceptor
  • Env vars for API keys — never hardcode

Advanced API Examples

// ── Retry with exponential backoff ───────────────────────────
async function fetchWithRetry(url, options = {}, retries = 3) {
  for (let attempt = 0; attempt < retries; attempt++) {
    try {
      const res = await fetch(url, options);
      if (res.status === 429 || res.status >= 500) {
        const delay = Math.min(1000 * 2 ** attempt, 30_000); // cap at 30s
        await new Promise(r => setTimeout(r, delay));
        continue;                   // retry
      }
      return res;
    } catch (err) {
      if (attempt === retries - 1) throw err; // last attempt — give up
      await new Promise(r => setTimeout(r, 1000 * 2 ** attempt));
    }
  }
}

// ── Client-side rate limiter ──────────────────────────────────
class RateLimiter {
  #queue = [];
  #running = 0;
  constructor(private maxConcurrent = 5) {}

  async run(fn) {
    while (this.#running >= this.maxConcurrent) {
      await new Promise(r => this.#queue.push(r));
    }
    this.#running++;
    try     { return await fn(); }
    finally {
      this.#running--;
      this.#queue.shift()?.();     // unblock next in queue
    }
  }
}
const limiter = new RateLimiter(5);
const results = await Promise.all(
  items.map(item => limiter.run(() => fetchItem(item)))
);

// ── CORS headers (Express server) ────────────────────────────
import cors from "cors";
app.use(cors({
  origin:      ["https://myapp.com", "http://localhost:3000"],
  methods:     ["GET","POST","PUT","PATCH","DELETE"],
  allowedHeaders: ["Content-Type","Authorization"],
  credentials: true,               // allow cookies/auth headers
}));

// Manual CORS for a single route
app.options("/api/*", (req, res) => {
  res.setHeader("Access-Control-Allow-Origin", "https://myapp.com");
  res.setHeader("Access-Control-Allow-Methods", "GET,POST");
  res.status(204).end();
});

// ── API key from env (server-side) ────────────────────────────
const OPENAI_KEY = process.env.OPENAI_API_KEY;
if (!OPENAI_KEY) throw new Error("Missing OPENAI_API_KEY");

const res = await fetch("https://api.openai.com/v1/chat/completions", {
  method:  "POST",
  headers: {
    "Authorization": `Bearer ${OPENAI_KEY}`,
    "Content-Type":  "application/json",
  },
  body: JSON.stringify({ model: "gpt-4o", messages: [...] }),
  signal: AbortSignal.timeout(30_000),
});

// ── Structured error from API ─────────────────────────────────
async function apiRequest(url, init) {
  const res = await fetch(url, init);
  if (res.ok) return res.json();
  const body = await res.json().catch(() => ({}));
  const err  = new Error(body.error ?? res.statusText);
  err.status = res.status;
  err.body   = body;
  throw err;
}

Notes

Exponential backoff — double the delay on each retry; cap at a maximum to avoid waiting minutes; jitter (random addition) prevents thundering herds
Rate limiter — controls max concurrent requests; prevents overwhelming an API or exhausting connection pools
CORS preflight — browsers send OPTIONS before cross-origin POST/PATCH/DELETE; your server must respond with 204 and the correct headers
credentials: true in CORS — required for cookies/auth headers across origins; the client must also set credentials: "include" in fetch
Never expose API keys to the browser — call third-party APIs from your server and proxy the response to the client
Attaching err.status and err.body to thrown errors allows callers to handle specific HTTP errors differently

Common Mistakes

  • Retrying on 4xx errors — only retry 429 (rate limit) and 5xx (server errors); 4xx are client errors that won't resolve on retry
  • Exposing third-party API keys to the browser — they appear in network requests and can be stolen; proxy through your server
  • Setting Access-Control-Allow-Origin: * with credentials: true — browsers reject this combination; must specify exact origin

Tips

  • Add jitter to retries: delay + Math.random() * 1000 — prevents multiple clients retrying in sync and overwhelming the server
  • Use AbortSignal.timeout(ms) on all external API calls — prevents hanging requests from blocking your server indefinitely
  • Log the X-Request-ID or correlation ID from API error responses — makes it possible to match your logs with the upstream provider's logs

Summary

  • Retry on 429/5xx with exponential backoff — never retry 4xx
  • Rate limit concurrent requests to avoid overwhelming APIs
  • CORS — configure origin, methods, credentials on the server · handle OPTIONS preflight
  • API keys in env vars · proxied server-side · never in browser code
