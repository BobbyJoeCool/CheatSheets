REST API Examples (Express)

// npm install express
import express from "express";
const app = express();
app.use(express.json());           // parse JSON bodies

// ── CRUD routes for /users ────────────────────────────────────
// GET /users?page=1&limit=10
app.get("/users", async (req, res) => {
  const page  = Number(req.query.page  ?? 1);
  const limit = Number(req.query.limit ?? 20);
  const users = await db.users.findMany({ skip: (page-1)*limit, take: limit });
  const total = await db.users.count();
  res.json({ data: users, meta: { page, limit, total } });
});

// GET /users/:id
app.get("/users/:id", async (req, res) => {
  const user = await db.users.findById(Number(req.params.id));
  if (!user) return res.status(404).json({ error: "User not found" });
  res.json(user);
});

// POST /users
app.post("/users", async (req, res) => {
  const parsed = CreateUserSchema.safeParse(req.body);
  if (!parsed.success) {
    return res.status(400).json({ errors: parsed.error.flatten().fieldErrors });
  }
  try {
    const user = await db.users.create(parsed.data);
    res.status(201)
       .location(`/users/${user.id}`)
       .json(user);
  } catch (err) {
    if (err.code === "23505") {     // unique constraint (postgres)
      return res.status(409).json({ error: "Email already exists" });
    }
    throw err;
  }
});

// PATCH /users/:id
app.patch("/users/:id", async (req, res) => {
  const user = await db.users.update(Number(req.params.id), req.body);
  if (!user) return res.status(404).json({ error: "Not found" });
  res.json(user);
});

// DELETE /users/:id
app.delete("/users/:id", async (req, res) => {
  await db.users.delete(Number(req.params.id));
  res.status(204).end();
});

// ── Global error handler ──────────────────────────────────────
app.use((err, req, res, next) => {
  console.error(err);
  res.status(500).json({ error: "Internal server error" });
});

app.listen(3000);

Notes

Return 201 Created + a Location header for POST — tells the client where to find the new resource
Return 204 No Content for DELETE — the resource is gone; no body to send
Validate with zod's safeParse in route handlers — return 400 with field errors, never let invalid data reach the database
Pagination — always paginate list endpoints; return meta with total, page, limit so clients know there's more
The global error handler must have (err, req, res, next) signature — Express identifies it by the 4-argument signature
Distinguish 401 (not authenticated) from 403 (authenticated but not authorised) — they signal different problems to the client

Common Mistakes

  • Returning 200 for POST creation — use 201 so clients know a resource was created
  • Returning 500 for validation errors — use 400 with field-level error details
  • Not adding a global error handler — unhandled errors leak stack traces to clients
  • Using GET with a body for filtering — use query parameters; GET bodies are not standard and often stripped by proxies

Tips

  • Use a consistent error response shape: { error: "message", errors: { field: ["msg"] } } — clients can parse it reliably
  • Validate and sanitise at the route boundary with zod — nothing untrusted should reach the database layer
  • Always paginate list endpoints from day one — retrofitting pagination is painful once clients depend on receiving all records

Summary

  • GET read · POST create (201) · PUT replace · PATCH partial · DELETE (204)
  • 400 validation · 401 unauthenticated · 403 forbidden · 404 not found · 409 conflict
  • Validate with zod at the route boundary · paginate all list endpoints
  • Global error handler catches unhandled errors · consistent error response shape
