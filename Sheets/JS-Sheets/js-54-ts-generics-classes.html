TS Generics in Classes

Generic classes · constrained generics · generic interfaces · factory patterns

Generic Classes

A generic class declares one or more type parameters that are shared across all its methods and properties. The type is fixed when the class is instantiated — new Stack<number>() creates a number-only stack, new Stack<string>() a string-only one. This gives you one reusable implementation with full type safety at every use site. Generic classes can have constraints (T extends Something) to restrict what types are allowed and gain access to specific properties or methods. Generic interfaces define contracts that generic classes can implement.

Key Points

  • Type param declared on the class: class Foo<T>
  • Type is fixed at instantiation: new Foo<string>()
  • All methods can use T from the class param
  • Methods can also add their own type params: method<U>()
  • Static members cannot use the class type param
  • Generic interfaces define typed contracts
  • Constraints limit and expose type capabilities

Generic Class Examples

// ── Generic Stack ─────────────────────────────────────────────
class Stack<T> {
  private items: T[] = [];

  push(item: T): void      { this.items.push(item); }
  pop(): T | undefined     { return this.items.pop(); }
  peek(): T | undefined    { return this.items.at(-1); }
  get size(): number       { return this.items.length; }
  get isEmpty(): boolean   { return this.items.length === 0; }

  // Method with its own type param (independent of T)
  map<U>(fn: (item: T) => U): Stack<U> {
    const result = new Stack<U>();
    this.items.forEach(item => result.push(fn(item)));
    return result;
  }
}

const numStack = new Stack<number>();
numStack.push(1); numStack.push(2); numStack.push(3);
numStack.pop();              // 3 — typed as number

const strStack = numStack.map(n => n.toString());
strStack.pop();              // "2" — typed as string

// ── Generic class with constraint ────────────────────────────
interface HasId {
  id: number;
}

class Repository<T extends HasId> {
  private store = new Map<number, T>();

  save(entity: T): void {
    this.store.set(entity.id, entity);  // ✔ T has .id
  }

  findById(id: number): T | undefined {
    return this.store.get(id);
  }

  findAll(): T[] {
    return [...this.store.values()];
  }

  delete(id: number): boolean {
    return this.store.delete(id);
  }
}

interface User { id: number; name: string; }
interface Product { id: number; price: number; }

const userRepo = new Repository<User>();
userRepo.save({ id: 1, name: "Alice" });
userRepo.findById(1)?.name;  // "Alice"

const productRepo = new Repository<Product>();
productRepo.save({ id: 10, price: 29.99 });

// ── Generic interface ─────────────────────────────────────────
interface Serializable<T> {
  serialize(): T;
  deserialize(data: T): void;
}

class JsonStore implements Serializable<string> {
  private data: Record<string, unknown> = {};

  serialize(): string { return JSON.stringify(this.data); }
  deserialize(json: string): void { this.data = JSON.parse(json); }
  set(key: string, val: unknown) { this.data[key] = val; }
}

// ── Factory pattern with generics ────────────────────────────
interface Constructor<T> {
  new (...args: unknown[]): T;
}

function create<T>(Ctor: Constructor<T>, ...args: unknown[]): T {
  return new Ctor(...args);
}

class Dog { constructor(public name: string) {} }
const rex = create(Dog, "Rex");
rex.name;  // "Rex" — typed as Dog

Notes

Class type param is fixed at instantiation — Stack<number> rejects strings at every push/pop call site
Individual methods can declare additional type params independent of the class param — useful for transformation methods like map<U>
Static members cannot use the class type param — statics belong to the class, not a particular instantiation; add a separate type param to the static method if needed
T extends HasId constraint — TS guarantees T has .id, so entity.id compiles without a cast
Constructor<T> — a type for constructor functions; the new (...args): T signature matches any class that produces a T
Generic repository pattern is ubiquitous in TS backends — one Repository<T> class serves all entity types

Common Mistakes

  • Using the class type param in a static member — TS errors; static members don't have access to instance type params
  • Forgetting to constrain when you need to access a property on T — TS assumes T could be anything without extends
  • Over-genericising — if a class only ever works with one type, use that type directly; generics add complexity

Tips

  • Use generic classes for data structures (Stack, Queue, Repository, Cache) — one implementation, infinite type-safe uses
  • Use T extends { id: number } constraints for CRUD repositories — clean and avoids casting
  • The Constructor<T> pattern is powerful for DI containers and factory functions — look it up when you need to work with class constructors as values

Summary

  • class Foo<T> — type fixed at instantiation; all methods share T
  • Methods can add independent type params: method<U>(fn: (t: T) => U): Foo<U>
  • T extends Constraint — restricts allowed types; grants access to constraint's members
  • Static members cannot use class type params — add a separate param to the static method instead
