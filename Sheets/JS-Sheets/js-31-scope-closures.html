Scope & Closures

Scope determines where a variable is accessible. JavaScript uses lexical (static) scoping — a function's scope is determined by where it is written, not where it is called. let and const are block-scoped ({ }); var is function-scoped. A closure is a function that retains access to its outer scope even after the outer function has returned. Closures are the mechanism behind private state, factory functions, event handlers, and memoisation. An IIFE (Immediately Invoked Function Expression) creates a private scope that runs once — less common in modern ES modules but still useful for isolation.

Scope Types

  • Global — accessible everywhere; avoid polluting
  • Module — file-level; not global in ESM
  • Function — inside a function; var lives here
  • Block — inside { }; let/const live here
  • Closure — function retains outer scope reference
  • Inner functions always see outer variables (scope chain)
  • Closures capture by reference, not by value

Scope & Closure Examples

// ── Block scope ───────────────────────────────────────────────
{
  let blockVar = "only here";
  const blockConst = 42;
}
console.log(blockVar);       // ✖ ReferenceError

// ── Function scope (var) ──────────────────────────────────────
function example() {
  var funcVar = "inside fn";
  if (true) {
    var funcVar = "still same var"; // same variable!
    let blockOnly = "in block";
  }
  console.log(funcVar);     // "still same var"
  console.log(blockOnly);   // ✖ ReferenceError
}

// ── Lexical scope — inner fn sees outer variables ─────────────
const outerVal = "outer";
function outer() {
  const mid = "mid";
  function inner() {
    console.log(outerVal);  // "outer"  — from global scope
    console.log(mid);       // "mid"    — from outer() scope
  }
  inner();
}

// ── Closure — retaining outer scope after function returns ────
function makeCounter(start = 0) {
  let count = start;             // private to makeCounter
  return {
    increment() { return ++count; },
    decrement() { return --count; },
    value()     { return count; },
  };
}
const c = makeCounter(10);
c.increment();                   // 11
c.increment();                   // 12
c.value();                       // 12
// count is NOT accessible from outside

// ── Factory function with closure ─────────────────────────────
function multiplier(factor) {
  return (n) => n * factor;      // closes over `factor`
}
const double = multiplier(2);
const triple = multiplier(3);
double(5);                       // 10
triple(5);                       // 15

// ── Classic closure bug — var in a loop ───────────────────────
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0); // 3 3 3  ← all share same i
}
// Fix: use let (block scope) instead
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0); // 0 1 2  ✔
}

// ── IIFE — Immediately Invoked Function Expression ────────────
const result = (function() {
  const private = "can't touch this";
  return { getValue: () => private };
})();
result.getValue();               // "can't touch this"

Notes

Closures capture variables by reference — they see the variable's current value at call time, not at creation time
The var loop bug — all callbacks share the same i variable; by the time they run, i is already at its final value
let in a loop — creates a new binding per iteration, so each closure captures its own copy
Factory functions use closures to create objects with private state — no class required
IIFE — wrapping in (function(){ })() creates a throw-away scope; useful for one-time initialisation in scripts
ES modules give every file its own scope — module-level variables are not global, reducing the need for IIFEs

Common Mistakes

  • Using var in loops with async callbacks — all share the same variable; use let
  • Expecting closures to capture values — they capture references; the value can change
  • Creating closures inside tight loops — each closure holds a reference to the outer scope, preventing garbage collection
  • Polluting global scope with var at the top level — use modules or block scope instead

Tips

  • Use closures for private state in factory functions — a clean alternative to classes for simple stateful objects
  • Use closures for memoisation: const memo = (() => { const cache = {}; return (k) => cache[k] ??= compute(k); })()
  • Always use let/const — block scoping eliminates the vast majority of scope-related bugs

Summary

  • Lexical scope — determined by where code is written, not where it runs
  • let/const are block-scoped · var is function-scoped (avoid)
  • Closure — function retains access to outer scope after outer function returns
  • Closures capture by reference · IIFE creates a private one-shot scope
