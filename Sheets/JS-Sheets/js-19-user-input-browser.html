Reading Input in the Browser

Browser user input comes from two main sources: the blocking prompt() dialog (simple but unusable in real apps) and HTML form elements — the proper approach. Form elements fire events (input, change, submit) that you listen to with addEventListener. Values are read from element.value (always a string — convert with Number() or parseInt() when needed). TypeScript requires casting DOM elements to their specific type (HTMLInputElement, HTMLSelectElement) to access .value and other properties.

Input Element Value Properties

  • <input type="text"> → .value (string)
  • <input type="number"> → .value (string!) · .valueAsNumber (number)
  • <input type="checkbox"> → .checked (boolean)
  • <input type="radio"> → .checked (boolean)
  • <input type="file"> → .files (FileList)
  • <select> → .value (selected option string)
  • <textarea> → .value (string)
  • .value is always a string — convert as needed

Browser Input Examples

// ── prompt() — simple but blocking (avoid in production) ─────
const name = prompt("Enter your name:");       // string | null
const age  = prompt("Enter your age:", "18");  // pre-filled
if (name !== null) console.log(`Hello, ${name}`);

// ── Reading an input element ─────────────────────────────────
const input = document.getElementById("username");
console.log(input.value);            // current value as string

// ── input event — fires on every keystroke ───────────────────
const field = document.getElementById("search");
field.addEventListener("input", (e) => {
  console.log(e.target.value);       // live value as user types
});

// ── change event — fires when focus leaves & value changed ───
const select = document.getElementById("color");
select.addEventListener("change", (e) => {
  console.log(e.target.value);       // selected option
});

// ── form submit — prevent default to handle in JS ───────────
const form = document.getElementById("myForm");
form.addEventListener("submit", (e) => {
  e.preventDefault();                // stop page reload
  const data = new FormData(form);
  console.log(data.get("username")); // value of name="username"
  console.log(data.get("email"));
});

// ── FormData — collect all fields at once ────────────────────
const fd = new FormData(form);
for (const [key, value] of fd.entries()) {
  console.log(key, value);           // each field name + value
}
const plain = Object.fromEntries(fd); // { username: "Alice", ... }

// ── TypeScript: type-cast DOM elements ───────────────────────
const input2 = document.getElementById("age") as HTMLInputElement;
const numVal: number = input2.valueAsNumber;   // ✔ typed number
const strVal: string = input2.value;           // ✔ typed string

const sel = document.querySelector("select") as HTMLSelectElement;
sel.addEventListener("change", (e: Event) => {
  const target = e.target as HTMLSelectElement;
  console.log(target.value);         // ✔ typed
});

// checkbox
const cb = document.getElementById("agree") as HTMLInputElement;
console.log(cb.checked);             // boolean

Notes

input event — fires on every character; use for live search/validation
change event — fires only when value changes and element loses focus (or on select/checkbox interaction)
e.preventDefault() on submit — stops the default browser behaviour of reloading the page
FormData — easiest way to collect all form fields at once; respects name attributes
.value is always a string — even type="number" inputs; use .valueAsNumber for a numeric value
TS: cast with as HTMLInputElement to access .value, .checked, .files etc. without type errors

Common Mistakes

  • Using prompt() in production apps — it blocks the page and is disabled in iframes and many contexts
  • Forgetting e.preventDefault() on form submit — causes page reload and data loss
  • Treating input.value as a number — it is always a string; parse it explicitly
  • Using e.target.value in TS without casting — EventTarget does not have .value

Tips

  • Use input.valueAsNumber for numeric inputs instead of Number(input.value) — cleaner and handles empty string as NaN
  • Debounce the input event for search fields — prevents firing an API call on every single keystroke
  • Use Object.fromEntries(new FormData(form)) to get a plain object of all form data in one line

Summary

  • prompt() — quick but blocking; avoid in real apps
  • input event — live updates · change event — on blur/commit
  • form submit → e.preventDefault() + FormData to read all fields
  • .value is always a string — use .valueAsNumber or parse explicitly
