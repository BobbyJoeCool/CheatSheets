Sheet 22

Switch & Pattern Matching

switch · case · default · break · fallthrough · lookup tables · TS discriminated unions

Switch Statements

switch compares one expression against multiple values using strict equality (===). Each case must end with break to prevent fallthrough — where execution continues into the next case. Fallthrough is occasionally intentional (grouping cases) but is more often a bug. The default clause handles unmatched values and is conventionally placed last. For many value-to-value mappings, an object lookup table is cleaner and more performant than a long switch. TypeScript's discriminated unions pair perfectly with switch — TS can verify all cases are handled and narrows the type in each branch.

Key Points

  • Uses strict equality (===) — no coercion
  • Always add break — fallthrough is usually a bug
  • Multiple cases can share a body (intentional fallthrough)
  • default runs if no case matches — put it last
  • Object lookup tables are often cleaner than long switches
  • TS checks exhaustiveness with never in default
  • return inside a case also prevents fallthrough

Switch & Lookup Examples

// ── Basic switch ─────────────────────────────────────────────
const day = "Monday";

switch (day) {
  case "Monday":
    console.log("Start of work week");
    break;
  case "Friday":
    console.log("End of work week");
    break;
  case "Saturday":
  case "Sunday":                  // intentional fallthrough — grouped
    console.log("Weekend!");
    break;
  default:
    console.log("Midweek");
}

// ── switch(true) — match conditions, not values ──────────────
const score = 78;
switch (true) {
  case score >= 90: console.log("A"); break;
  case score >= 80: console.log("B"); break;
  case score >= 70: console.log("C"); break;
  default:          console.log("F");
}                               // "C"

// ── Object lookup table — often cleaner than switch ──────────
const httpStatus = {
  200: "OK",
  201: "Created",
  400: "Bad Request",
  401: "Unauthorized",
  404: "Not Found",
  500: "Internal Server Error",
};
const msg = httpStatus[404] ?? "Unknown";  // "Not Found"

// ── Function dispatch table ───────────────────────────────────
const actions = {
  increment: (n) => n + 1,
  decrement: (n) => n - 1,
  double:    (n) => n * 2,
};
const result = actions["double"](5);  // 10

// ── TypeScript: discriminated union + exhaustive switch ───────
type Shape =
  | { kind: "circle";   radius: number }
  | { kind: "square";   side: number }
  | { kind: "rectangle"; width: number; height: number };

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "square":
      return shape.side ** 2;
    case "rectangle":
      return shape.width * shape.height;
    default:
      const _exhaustive: never = shape; // TS error if case missed
      return _exhaustive;
  }
}

Notes

Forgetting break causes silent fallthrough — execution continues into the next case's body
switch(true) — evaluates each case as a boolean; useful for range checks without deep nesting
Object lookup tables have O(1) lookup vs O(n) for a long switch chain; cleaner for static mappings
Function dispatch tables allow runtime selection of behaviour — common in state machines and command patterns
Discriminated unions — TS narrows shape in each case to its specific type; accessing wrong fields is a compile error
The never exhaustiveness check — if a new Shape variant is added and not handled, default catches it and TS errors

Common Mistakes

  • Forgetting break — the most common switch bug; execution silently falls through to the next case
  • Using switch with loose types — it uses ===, so "1" and 1 are different cases
  • Omitting default — unmatched values pass silently with no indication
  • Writing long switches for simple value mappings — an object lookup is shorter and clearer

Tips

  • Use return inside switch cases in functions — it implicitly prevents fallthrough and eliminates break
  • For 5+ cases that map a value to another value, use an object lookup table — it is more readable and easier to maintain
  • Always add the never exhaustiveness check in TypeScript discriminated union switches — it is free compile-time safety

Summary

  • switch uses === — always add break or return to prevent fallthrough
  • Group cases with intentional fallthrough — case "a": case "b": doThing()
  • Object lookup tables — cleaner than long switches for static value mappings
  • TS discriminated unions + never default — exhaustive, type-safe switching
