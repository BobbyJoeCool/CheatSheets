Crypto & Security

Node Cryptography

Node's built-in crypto module provides hashing, HMAC, symmetric encryption, key generation, and secure random bytes. For password storage never use SHA/MD5 — use bcrypt or Argon2 which are slow by design and resistant to brute-force. For API authentication, JWT (JSON Web Tokens) are the standard stateless token format. For data in transit, TLS handles encryption — you rarely need to implement it yourself. The Web Crypto API (crypto.subtle) is now available in Node 19+ for browser-compatible cryptographic operations.

Algorithm Guide

  • SHA-256 — checksums, content IDs, non-secret hashes
  • HMAC-SHA256 — message authentication, API signatures
  • AES-256-GCM — symmetric encryption (authenticated)
  • bcrypt — password hashing; never SHA for passwords
  • Argon2 — stronger than bcrypt (argon2 package)
  • JWT — stateless auth tokens (jsonwebtoken / jose)
  • Use crypto.randomBytes() for tokens, not Math.random()

Crypto & Security Examples

import crypto from "crypto";

// ── Hashing — SHA-256 ────────────────────────────────────────
function sha256(data) {
  return crypto.createHash("sha256").update(data).digest("hex");
}
sha256("hello");  // "2cf24dba5fb0a30e..."

// Hash a file
import { createReadStream } from "fs";
async function hashFile(path) {
  const hash   = crypto.createHash("sha256");
  const stream = createReadStream(path);
  for await (const chunk of stream) hash.update(chunk);
  return hash.digest("hex");
}

// ── HMAC — signed hash ────────────────────────────────────────
const SECRET = process.env.HMAC_SECRET;
function sign(data) {
  return crypto.createHmac("sha256", SECRET).update(data).digest("hex");
}
function verify(data, signature) {
  const expected = sign(data);
  return crypto.timingSafeEqual(            // constant-time comparison
    Buffer.from(signature, "hex"),
    Buffer.from(expected,  "hex")
  );
}

// ── AES-256-GCM — symmetric encryption ───────────────────────
const ALGO = "aes-256-gcm";
const KEY  = crypto.scryptSync(process.env.SECRET_KEY, "salt", 32); // 256-bit

function encrypt(plaintext) {
  const iv         = crypto.randomBytes(12);        // 96-bit IV for GCM
  const cipher     = crypto.createCipheriv(ALGO, KEY, iv);
  const encrypted  = Buffer.concat([
    cipher.update(plaintext, "utf8"),
    cipher.final(),
  ]);
  const authTag = cipher.getAuthTag();              // 16-byte auth tag
  return { iv: iv.toString("hex"), encrypted: encrypted.toString("hex"),
           authTag: authTag.toString("hex") };
}

function decrypt({ iv, encrypted, authTag }) {
  const decipher = crypto.createDecipheriv(
    ALGO, KEY, Buffer.from(iv, "hex")
  );
  decipher.setAuthTag(Buffer.from(authTag, "hex"));
  return decipher.update(encrypted, "hex", "utf8") + decipher.final("utf8");
}

// ── Secure random tokens ──────────────────────────────────────
const token  = crypto.randomBytes(32).toString("hex");       // 64-char hex
const urlSafe= crypto.randomBytes(32).toString("base64url"); // base64url
const uuid   = crypto.randomUUID();

// ── bcrypt — password hashing ─────────────────────────────────
// npm install bcrypt  (+ @types/bcrypt for TS)
import bcrypt from "bcrypt";

const SALT_ROUNDS = 12;  // higher = slower = more secure

// Hash on registration
const hash = await bcrypt.hash(plainPassword, SALT_ROUNDS);

// Verify on login
const isValid = await bcrypt.compare(inputPassword, storedHash);

// ── JWT — JSON Web Tokens ─────────────────────────────────────
// npm install jsonwebtoken  (+ @types/jsonwebtoken)
import jwt from "jsonwebtoken";

const JWT_SECRET = process.env.JWT_SECRET;

// Sign — on login
const token2 = jwt.sign(
  { userId: 1, role: "admin" },    // payload
  JWT_SECRET,
  { expiresIn: "7d", algorithm: "HS256" }
);

// Verify — on each request
try {
  const payload = jwt.verify(token2, JWT_SECRET);
  console.log(payload.userId, payload.role);
} catch (err) {
  // TokenExpiredError, JsonWebTokenError
  res.status(401).json({ error: "Invalid token" });
}

Notes

Never use SHA/MD5 for passwords — they're too fast; attackers can try billions/second; bcrypt/Argon2 are deliberately slow
crypto.timingSafeEqual() — prevents timing attacks where attackers deduce a secret by measuring how long comparison takes
AES-GCM provides both encryption and authentication (tamper detection) — always prefer GCM over plain CBC mode
The IV (initialisation vector) must be random and unique per encryption — never reuse an IV with the same key
bcrypt SALT_ROUNDS = 12 is the current recommended minimum — increase over time as hardware gets faster
JWT payload is signed, not encrypted — anyone can decode it; never put secrets or sensitive data in the payload

Common Mistakes

  • Hashing passwords with SHA256 — use bcrypt or Argon2; SHA is designed to be fast, which makes it terrible for passwords
  • Using === for comparing HMACs/hashes — vulnerable to timing attacks; use crypto.timingSafeEqual()
  • Putting secrets in JWT payload — it's only base64-encoded, not encrypted; anyone can decode it
  • Reusing IVs in AES encryption — catastrophic for security; always generate a fresh random IV per encryption

Tips

  • Use jose (npm) instead of jsonwebtoken for modern JWT handling — Web Crypto based, works in Edge runtimes, supports RS256
  • Store the JWT secret in an env var with at least 256 bits of entropy: node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
  • Rotate encryption keys by keeping a key ID with encrypted data — allows decrypting old data while encrypting new data with a new key

Summary

  • SHA-256 for checksums · HMAC-SHA256 for message auth · AES-256-GCM for encryption
  • bcrypt/Argon2 for passwords — never SHA · always timingSafeEqual for hash comparison
  • JWT: signed not encrypted · payload is public · sign with HS256/RS256
  • crypto.randomBytes(32) for secure tokens · unique random IV per AES encryption
