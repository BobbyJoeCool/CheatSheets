Sheet 79

Event Loop

Call stack · task queue · microtask queue · setTimeout · queueMicrotask · execution order

Execution Contexts & Queues

ContextWhat Goes HerePriority
Call StackSynchronous code currently executingRuns first — always clears before anything else
Microtask QueuePromise callbacks (.then/.catch), queueMicrotask(), MutationObserverRuns after each task, before next task — drains completely
Macrotask QueuesetTimeout, setInterval, I/O callbacks, UI eventsOne task per event loop iteration
Animation QueuerequestAnimationFrameBefore each paint (browser only)

Event Loop Examples

// ── Execution order demonstration ─────────────────────────────
console.log("1: sync start");

setTimeout(() => console.log("5: macrotask (setTimeout 0)"), 0);

Promise.resolve()
  .then(() => console.log("3: microtask (Promise)"))
  .then(() => console.log("4: microtask (chained Promise)"));

queueMicrotask(() => console.log("3b: microtask (queueMicrotask)"));

console.log("2: sync end");

// Output order:
// 1: sync start
// 2: sync end
// 3: microtask (Promise)       ← microtasks run before macrotasks
// 3b: microtask (queueMicrotask)
// 4: microtask (chained Promise)
// 5: macrotask (setTimeout 0)  ← setTimeout fires AFTER microtasks

// ── setTimeout / setInterval ──────────────────────────────────
const id = setTimeout(() => {
  console.log("Runs once after ~1000ms");
}, 1000);

clearTimeout(id);                // cancel before it fires

const interval = setInterval(() => {
  console.log("Runs every 500ms");
}, 500);

setTimeout(() => clearInterval(interval), 2500); // stop after ~2.5s

// ── Minimum delay caveat ──────────────────────────────────────
// setTimeout(fn, 0) does NOT run immediately
// It queues a macrotask — runs after the current task + all microtasks
setTimeout(() => console.log("Not immediate!"), 0);
Promise.resolve().then(() => console.log("I run first"));

// ── Blocking the event loop ───────────────────────────────────
// ✖ This blocks all I/O, timers, and user events for 5 seconds:
const start = Date.now();
while (Date.now() - start < 5000) {} // NEVER do this in Node

// ✔ Use async operations instead:
await new Promise(r => setTimeout(r, 5000)); // yields to event loop

// ── queueMicrotask ────────────────────────────────────────────
// Schedule a callback as a microtask (before setTimeout, after sync)
queueMicrotask(() => {
  console.log("Runs after current sync, before next macrotask");
});

// ── Node.js: process.nextTick ────────────────────────────────
// Runs BEFORE Promise microtasks (even higher priority)
process.nextTick(() => console.log("nextTick — before Promise.then"));
Promise.resolve().then(() => console.log("Promise.then"));
// nextTick runs first

// ── setImmediate (Node only) ──────────────────────────────────
setImmediate(() => {
  console.log("Runs after I/O events in Node event loop");
});

Notes

Event loop order: call stack → microtasks (drain completely) → one macrotask → repeat
Microtask queue drains completely between macrotasks — if a microtask queues another microtask, it runs in the same drain cycle
setTimeout(fn, 0) — minimum ~4ms delay in browsers; actual delay depends on system load and timer resolution
Blocking the call stack (CPU-bound loops) freezes the entire event loop — no timers, no I/O, no user input can be processed
Node's process.nextTick runs before Promise microtasks — it was added before the microtask queue was standardised; prefer queueMicrotask
Understanding the event loop is essential for debugging unexpected ordering in async code and performance issues in Node servers

Common Mistakes

  • Assuming setTimeout(fn, 0) runs immediately — it queues a macrotask; Promises run before it
  • Running CPU-intensive work synchronously in Node — blocks all other connections for the duration
  • Using process.nextTick recursively — can starve the microtask and macrotask queues

Tips

  • Move CPU-heavy work to a Worker Thread (Node) or Web Worker (browser) — keeps the event loop free
  • Use setImmediate (Node) to yield to I/O between chunks of CPU work: await new Promise(r => setImmediate(r))
  • Use queueMicrotask(fn) to schedule work after current sync but before timers — preferred over Promise.resolve().then(fn)

Summary

  • Order: sync code → microtasks (drain) → one macrotask → repeat
  • Microtasks: Promise .then · queueMicrotask — run before any setTimeout
  • Macrotasks: setTimeout · setInterval · I/O callbacks — one per loop iteration
  • Never block the call stack — use async operations to yield to the event loop
