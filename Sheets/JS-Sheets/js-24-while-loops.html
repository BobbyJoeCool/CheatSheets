Sheet 24

While Loops

while · do...while · infinite loops · polling patterns · loop guards

while & do...while

while loops repeat as long as a condition is truthy, checking before each iteration. If the condition is false at the start, the body never runs. do...while is identical except the condition is checked after the body — guaranteeing the body runs at least once. Use while when the iteration count is unknown and depends on runtime state: reading until EOF, polling until ready, processing a queue until empty. The classic for loop is preferred when the iteration count is known in advance. Always ensure the loop condition can eventually become false — forgetting to update the condition variable is the most common cause of infinite loops.

while vs do...while

  • while — checks condition before each iteration
  • do...while — checks condition after each iteration
  • do...while body always runs at least once
  • Both support break and continue
  • Use while(true) + break for explicit infinite loops
  • Always update the condition variable inside the loop body
  • while is ideal for queue/stack drain patterns

While Loop Examples

// ── Basic while ──────────────────────────────────────────────
let count = 0;
while (count < 5) {
  console.log(count);           // 0 1 2 3 4
  count++;                      // must update or infinite loop!
}

// ── do...while — body runs at least once ─────────────────────
let input;
do {
  input = prompt("Enter a number greater than 0:");
  input = Number(input);
} while (input <= 0 || isNaN(input));
console.log("You entered:", input);

// ── Draining a queue ─────────────────────────────────────────
const queue = [1, 2, 3, 4, 5];
while (queue.length > 0) {
  const item = queue.shift();   // remove first element
  console.log("Processing:", item);
}

// ── while(true) with break ───────────────────────────────────
let attempts = 0;
while (true) {
  const result = tryConnect();
  if (result.ok) break;         // exit on success
  attempts++;
  if (attempts >= 3) {
    throw new Error("Max retries exceeded");
  }
}

// ── Polling pattern ──────────────────────────────────────────
async function waitForReady(resource) {
  while (!resource.isReady()) {
    await sleep(500);            // wait 500ms between checks
  }
  return resource;
}
const sleep = (ms) => new Promise(r => setTimeout(r, ms));

// ── Processing linked list ────────────────────────────────────
let node = headNode;
while (node !== null) {
  console.log(node.value);
  node = node.next;
}

// ── do...while for menu loop ─────────────────────────────────
let choice;
do {
  choice = showMenu();          // always show menu at least once
  handleChoice(choice);
} while (choice !== "quit");

Notes

while — if the condition is false on the first check, the body never executes
do...while — the semicolon after } is required: } while (condition);
Infinite loop safety — always ensure the loop has a guaranteed exit: updating a counter, a break, or an async condition
queue.shift() in a while loop — classic queue drain; stack.pop() drains a stack
Async polling with while (!ready) — always add a sleep() delay and a max-attempts guard to avoid hammering a resource
Linked list traversal is one of the clearest use cases for while (node !== null)

Common Mistakes

  • Forgetting to update the condition variable — creates an infinite loop that freezes the process
  • Missing the semicolon on do...while: } while(cond) — it is required
  • Polling without a delay — while(!ready){} with no await sleep() blocks the event loop entirely
  • Using while when a for loop is clearer — if the count is known, use for

Tips

  • Add a max-iterations guard to any while(true) loop — prevents runaway loops in production
  • Use do...while for input validation loops — you always need to prompt at least once before you can check the result
  • For async polling, combine while + await sleep() + a timeout/abort signal to avoid hanging indefinitely

Summary

  • while (cond) — checks before; body may never run if condition starts false
  • do { } while (cond); — checks after; body always runs at least once
  • Always ensure the exit condition can be reached — update the variable or use break
  • Best for: queue drain · polling · unknown iteration count · linked list traversal
