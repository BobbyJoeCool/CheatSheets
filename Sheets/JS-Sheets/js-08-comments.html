Sheet 08

Comments

Single-line · block · inline · TODO/FIXME · best practices

About Comments

Comments are ignored by the JavaScript engine — they exist purely for humans. JS/TS has two syntaxes: single-line (//) and block (/* */). A third form, JSDoc (/** */), is a convention that tooling (VS Code, TypeDoc, JSDoc) parses to generate documentation and IntelliSense hints — covered on Sheet 09. The golden rule: comments should explain why, not what. If you need a comment to explain what code does, consider whether the code itself could be clearer first.

Comment Types

  • // text — single-line; runs to end of line
  • /* text */ — block; spans multiple lines
  • /** text */ — JSDoc; parsed by tooling (Sheet 09)
  • Cannot nest block comments: /* /* */ */ breaks
  • Single-line comments can appear at end of a line (inline)
  • Block comments can appear mid-expression (unusual but valid)

Comment Syntax & Conventions

// ── Single-line comment ──────────────────────────────────────
let x = 5;                            // inline — explains this line

// Full-line comment — use for brief explanations above code
// Multiple single-line comments are fine for short paragraphs

// ── Block comment ────────────────────────────────────────────
/*
  Block comments span multiple lines.
  Use for longer explanations or section headers.
  Also used to temporarily disable code.
*/

/* Can also be written on one line */

// ── Disabling code ───────────────────────────────────────────
// console.log("disabled temporarily");  ← single line

/*
fetch("/api/data")
  .then(r => r.json())
  .then(console.log);
*/                                    // ← whole block disabled

// ── Section dividers (common in large files) ─────────────────
// ════════════════════════════════════
// SECTION: Utility Functions
// ════════════════════════════════════

// ── TODO / FIXME / HACK / NOTE tags ─────────────────────────
// TODO: add input validation before shipping
// FIXME: crashes when value is null — see issue #42
// HACK: temporary workaround until the API v2 ships
// NOTE: order matters here — do not reorder these calls
// PERF: this loop is O(n²) — optimise if dataset > 1000

// ── TypeScript-specific comment directives ───────────────────
// @ts-ignore          ← suppress the NEXT line's TS error
// @ts-expect-error    ← like ts-ignore but errors if no error (preferred)
// @ts-nocheck         ← disable TS checking for the entire file
// @ts-check           ← enable TS checking in a plain .js file

// @ts-expect-error — next line intentionally passes wrong type
doThing(42 as unknown as string);

// ── Inline block comment (unusual but valid) ─────────────────
const result = /* mid-expression comment */ getValue();

Notes

// — preferred for most comments; easy to toggle with Ctrl+/ in VS Code
/* */ — cannot be nested; use for multi-line disabling or longer prose blocks
TODO/FIXME tags are highlighted by VS Code extensions and caught by ESLint plugins
@ts-expect-error is preferred over @ts-ignore — it errors if the suppression is no longer needed
@ts-check at the top of a .js file enables TS type inference without a tsconfig
Dead code is better deleted than commented out — git history is your undo

Common Mistakes

  • Comments that restate the code: // increment x by 1 above x++ — noise, not signal
  • Leaving large commented-out blocks in the codebase — use git branches instead
  • Using @ts-ignore instead of @ts-expect-error — the latter keeps you honest
  • Trying to nest block comments: /* outer /* inner */ */ — the first */ closes both

Tips

  • Install Todo Tree in VS Code to visualise all TODO/FIXME tags across a project
  • Use eslint-plugin-no-warning-comments to fail CI if TODOs are left in release branches
  • Write comments before you write the code — forces you to think through the logic first

Summary

  • // single-line · /* */ block · /** */ JSDoc (Sheet 09)
  • Comments explain why, not what — the code shows what
  • TODO/FIXME/HACK/NOTE — standard tags recognised by editors and linters
  • @ts-expect-error — preferred TS directive to suppress a known type error
