Async / Await

async/await is syntactic sugar over Promises — it makes asynchronous code look and read like synchronous code. An async function always returns a Promise. await pauses execution until the awaited Promise settles, then unwraps the value. Errors from rejected Promises become thrown exceptions that try/catch handles naturally. The key trap is sequential vs parallel execution — awaiting two independent operations one after the other is slower than running them in parallel with Promise.all.

Key Rules

  • async functions always return a Promise
  • await can only be used inside async functions (or top-level ESM)
  • Awaiting a non-Promise value just returns it
  • Rejected Promises become thrown errors
  • Sequential await = serial; use Promise.all for parallel
  • TS: async fn(): Promise<T> — return type is always Promise<T>
  • Top-level await works in ESM files (Node 14.8+)

Async/Await Examples

// ── Basic async function ──────────────────────────────────────
async function getUser(id) {
  const res  = await fetch(`/api/users/${id}`);
  const user = await res.json();
  return user;                       // implicitly wrapped in Promise
}

// Returns a Promise — both work:
getUser(1).then(console.log);
const user = await getUser(1);       // inside another async fn

// ── Error handling ────────────────────────────────────────────
async function fetchSafe(url) {
  try {
    const res = await fetch(url);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return await res.json();
  } catch (err) {
    console.error("Fetch failed:", err.message);
    return null;
  }
}

// ── Sequential (slow) vs parallel (fast) ─────────────────────
// ✖ Sequential — total time = A + B + C
async function sequential() {
  const a = await fetchA();          // wait for A
  const b = await fetchB();          // then wait for B
  const c = await fetchC();          // then wait for C
  return { a, b, c };
}

// ✔ Parallel — total time = max(A, B, C)
async function parallel() {
  const [a, b, c] = await Promise.all([fetchA(), fetchB(), fetchC()]);
  return { a, b, c };
}

// ✔ Start all, await separately (same as Promise.all)
async function parallelAlt() {
  const pA = fetchA();               // start immediately (no await)
  const pB = fetchB();
  const pC = fetchC();
  return { a: await pA, b: await pB, c: await pC };
}

// ── Async in loops ────────────────────────────────────────────
// ✖ forEach doesn't await — all run concurrently (uncontrolled)
items.forEach(async item => { await process(item); });

// ✔ Sequential processing with for...of
for (const item of items) {
  await process(item);               // processes one at a time
}

// ✔ Parallel with concurrency control (process N at a time)
const BATCH = 5;
for (let i = 0; i < items.length; i += BATCH) {
  await Promise.all(items.slice(i, i + BATCH).map(process));
}

// ── TypeScript: typed async functions ────────────────────────
async function fetchUser(id: number): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json() as Promise<User>;
}

// Awaited utility type — get the resolved type
type UserResult = Awaited<ReturnType<typeof fetchUser>>; // User

// ── Top-level await (ESM only) ────────────────────────────────
const config = await loadConfig();   // no async wrapper needed
const db     = await connectDb(config.dbUrl);

Notes

An async function that returns a value wraps it in Promise.resolve(); throwing wraps in Promise.reject()
await unwraps nested Promises — await Promise.resolve(Promise.resolve(42)) yields 42
The sequential vs parallel trap — always ask: "do these operations depend on each other?" If not, use Promise.all
forEach with async — the callback returns a Promise that forEach ignores; the loop finishes before any work completes
Batch processing — Promise.all with full array can overwhelm a server; batching limits concurrent requests to N at a time
TS Awaited<T> — unwraps Promise<Promise<User>> all the way to User

Common Mistakes

  • Sequential await for independent operations — adds latency unnecessarily; use Promise.all
  • Using async in forEach — the loop doesn't wait; use for...of or Promise.all(arr.map(async ...))
  • Forgetting that unhandled async errors still crash/silently fail — always wrap in try/catch or attach .catch()
  • Using top-level await in a CommonJS file — only works in ESM; throws a SyntaxError

Tips

  • Start independent Promises before awaiting them: const p1 = fetch(a); const p2 = fetch(b); await p1; await p2 — both run in parallel
  • Use for...of with await for sequential processing where order matters
  • Use Promise.all(arr.map(async item => ...)) to map an array of async operations in parallel

Summary

  • async always returns a Promise · await unwraps it · rejected Promises throw
  • Sequential await = serial · Promise.all([...]) = parallel
  • Never use async in forEach — use for...of or Promise.all(arr.map(...))
  • Top-level await in ESM files — no async wrapper needed
