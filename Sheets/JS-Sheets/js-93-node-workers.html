Worker Threads

Node.js is single-threaded — CPU-intensive work blocks the event loop and stalls all concurrent requests. Worker threads run JavaScript in a separate V8 instance with its own event loop, allowing true parallelism on multi-core machines. Workers share memory address space (unlike child processes) and can exchange data via message passing or SharedArrayBuffer for zero-copy transfer. Use workers for: image processing, cryptography, data transformation, compression, and any task that keeps the CPU busy for more than a few milliseconds.

Key APIs

  • new Worker(filename, { workerData }) — spawn worker
  • worker.postMessage(data) — send data to worker
  • worker.on("message", cb) — receive from worker
  • worker.on("error", cb) — worker threw an error
  • worker.terminate() — stop worker immediately
  • parentPort.postMessage(data) — worker → main
  • workerData — initial data passed at spawn time
  • isMainThread — true if not in a worker

Worker Thread Examples

// ── worker.js — runs in the worker thread ─────────────────────
import { parentPort, workerData } from "worker_threads";

function heavyComputation(n) {
  let result = 0;
  for (let i = 0; i < n; i++) result += Math.sqrt(i);
  return result;
}

const result = heavyComputation(workerData.iterations);
parentPort.postMessage({ result });

// ── main.js — spawns the worker ───────────────────────────────
import { Worker } from "worker_threads";
import { fileURLToPath } from "url";
import path from "path";

const __dirname = path.dirname(fileURLToPath(import.meta.url));

function runWorker(workerData) {
  return new Promise((resolve, reject) => {
    const worker = new Worker(
      path.join(__dirname, "worker.js"),
      { workerData }
    );
    worker.on("message", resolve);
    worker.on("error",   reject);
    worker.on("exit", code => {
      if (code !== 0) reject(new Error(`Worker exited: ${code}`));
    });
  });
}

const { result } = await runWorker({ iterations: 1_000_000 });
console.log("Result:", result);

// ── Two-way messaging ─────────────────────────────────────────
// worker.js
import { parentPort } from "worker_threads";
parentPort.on("message", ({ task, data }) => {
  if (task === "process") {
    const output = data.map(n => n * 2);
    parentPort.postMessage({ task, output });
  }
});

// main.js
const worker = new Worker("./worker.js");
worker.postMessage({ task: "process", data: [1, 2, 3] });
worker.on("message", ({ output }) => console.log(output)); // [2, 4, 6]

// ── Worker pool — reuse workers ───────────────────────────────
class WorkerPool {
  #workers = [];
  #queue   = [];

  constructor(filename, size = 4) {
    for (let i = 0; i < size; i++) {
      const w = new Worker(filename);
      w.on("message", (result) => {
        const { resolve } = w._task;
        w._task = null;
        this.#idle(w);
        resolve(result);
      });
      w.on("error", (err) => { w._task?.reject(err); });
      this.#workers.push(w);
    }
  }

  run(data) {
    return new Promise((resolve, reject) => {
      const free = this.#workers.find(w => !w._task);
      if (free) { this.#dispatch(free, data, resolve, reject); }
      else       { this.#queue.push({ data, resolve, reject }); }
    });
  }

  #dispatch(w, data, resolve, reject) {
    w._task = { resolve, reject };
    w.postMessage(data);
  }

  #idle(w) {
    const next = this.#queue.shift();
    if (next) this.#dispatch(w, next.data, next.resolve, next.reject);
  }
}

// ── SharedArrayBuffer — zero-copy shared memory ───────────────
const shared  = new SharedArrayBuffer(4);
const counter = new Int32Array(shared);

const w = new Worker("./counter-worker.js", {
  workerData: { shared }
});
// Both main and worker share the same memory buffer
Atomics.add(counter, 0, 1);       // thread-safe increment

Notes

Workers start their own V8 instance — startup cost (~10ms) means they're best for long-running tasks, not tiny computations
Message passing uses the structured clone algorithm — copies data (including Maps, Sets, Dates); use transferList for zero-copy transfer of ArrayBuffers
Worker pool — spawning a new worker per task is wasteful; a pool of N workers (N ≈ CPU count) processes tasks from a queue efficiently
SharedArrayBuffer — both threads access the same memory; always use Atomics for thread-safe reads/writes to avoid data races
workerData is passed at creation time and is immutable from the worker's perspective — use postMessage for dynamic data exchange
Use os.cpus().length to determine pool size — match workers to available CPU cores

Common Mistakes

  • Spawning a new Worker for every request — the startup cost negates the benefit; use a pool
  • Not handling "error" and "exit" events — failed workers leak resources silently
  • Using SharedArrayBuffer without Atomics — race conditions produce unpredictable results
  • Using workers for I/O-bound tasks — workers help CPU-bound work only; I/O is already async via the event loop

Tips

  • Use os.cpus().length for pool size — one worker per logical CPU core is the standard starting point
  • Transfer ArrayBuffers instead of copying: worker.postMessage(buffer, [buffer]) — zero-copy, the main thread loses access
  • Consider piscina (npm) — a battle-tested worker pool implementation with TypeScript support; saves writing your own pool

Summary

  • Workers run JS in parallel threads — use for CPU-bound work that blocks the event loop
  • workerData — initial data · postMessage — two-way messaging · structured clone copies data
  • Worker pool — reuse N workers (N = CPU count) for a queue of tasks
  • SharedArrayBuffer + Atomics — zero-copy shared memory with thread safety
