Sheet 04

Strict Mode & Boilerplate

"use strict" · what it prevents · module auto-strict · common entry patterns

What is Strict Mode?

Strict mode is an opt-in variant of JavaScript that eliminates some silent errors by throwing them instead, disallows syntax likely to be defined in future ECMAScript versions, and sometimes runs faster because engines can perform more optimisations. Activate it by placing "use strict"; at the very top of a script file or function body. ES Modules are always in strict mode automatically — you never need to write "use strict" in a module file. TypeScript files compiled with strict: true in tsconfig also enforce strict-mode semantics and much more.

What Strict Mode Prevents

  • Using undeclared variables — throws ReferenceError
  • Writing to read-only properties — throws TypeError
  • Deleting non-deletable properties — throws TypeError
  • Duplicate parameter names in functions
  • Using with statement (deprecated)
  • this is undefined in regular functions (not window)
  • Octal numeric literals (0755)

Strict Mode & Entry Point Patterns

// ── Enabling strict mode (script files only) ─────────────────
"use strict";                         // must be the FIRST statement

x = 5;                                // ✖ ReferenceError (undeclared)
let x = 5;                            // ✔ correct
delete Object.prototype;              // ✖ TypeError
function f(a, a) { }                  // ✖ SyntaxError: duplicate param

// ── Modules are always strict — no declaration needed ────────
// In any file loaded as a module (type="module" or .mjs):
badVar = 10;                          // ✖ ReferenceError — strict is on

// ── Function-level strict mode ───────────────────────────────
function legacyCode() {
  "use strict";                       // strict only inside this function
  undeclared = 1;                     // ✖ ReferenceError
}

// ── Common Node.js boilerplate patterns ──────────────────────

// Pattern 1: simple script
console.log("Hello, Node!");

// Pattern 2: async wrapper (works in any Node version)
async function main() {
  const result = await doSomething();
  console.log(result);
}
main().catch(console.error);          // always catch top-level errors

// Pattern 3: top-level await (ESM, Node 14.8+)
// No wrapper needed — await can be used at top level
const config = await loadConfig();    // only in ESM (.mjs or type:module)

// Pattern 4: TypeScript entry point
import { serve } from "./server.js";

async function main(): Promise<void> {
  await serve({ port: 3000 });
  console.log("Listening on :3000");
}
main().catch((err: unknown) => {
  console.error(err);
  process.exit(1);
});

Notes

"use strict" must be a literal string — the very first statement (before any code)
Comments before "use strict" are fine; any other statement disables it
ES Modules and TypeScript enforce strict automatically — no opt-in needed
main().catch(console.error) is the safest catch-all for unhandled async errors
TypeScript's strict: true includes noImplicitAny, strictNullChecks, and 6 other checks
process.exit(1) signals failure to the OS — important for CI/CD pipelines

Common Mistakes

  • Placing "use strict" after other code — it silently does nothing
  • Writing "use strict" in module files — redundant, but harmless
  • Forgetting .catch() on the main() call — unhandled promise rejections crash Node 15+
  • Using top-level await in a CommonJS file — throws a SyntaxError

Tips

  • Just use ESM or TypeScript — strict mode is then automatic and you never think about it again
  • In TypeScript, strict: true in tsconfig is far more powerful than JS's "use strict"
  • Add process.on("unhandledRejection", ...) as a global safety net in long-running Node servers

Summary

  • "use strict" — turns silent JS errors into thrown errors; must be first statement
  • ES Modules and TypeScript are always strict — no opt-in required
  • JS has no required entry point — execution is top-to-bottom
  • Wrap async code in main().catch() or use top-level await (ESM only)
