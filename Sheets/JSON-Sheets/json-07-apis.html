Sheet 07

JSON in APIs & HTTP

REST conventions · headers · request/response · pagination · error patterns

JSON as the API Wire Format

JSON is the de-facto wire format for REST and most HTTP APIs. The client signals it is sending JSON with Content-Type: application/json and requests JSON back with Accept: application/json. Well-designed APIs return consistent envelope shapes — a data wrapper for success responses and a structured error object for failures — so clients can handle both cases with the same parsing logic.

HTTP Header Rules

  • Content-Type: application/json — set on any request body you send
  • Accept: application/json — tells the server what format you want back
  • Content-Type: application/json; charset=utf-8 — explicit encoding (usually redundant)
  • GET requests never have a body — pass data in query params, not JSON
  • 201 Created should return the newly created resource in the body
  • 204 No Content returns an empty body — don't try to parse it

Common API JSON Shapes

// ── Success: single resource ──────────────────────────────────
{
  "data": { "id": 1, "name": "Alice", "email": "alice@example.com" }
}

// ── Success: collection with pagination ───────────────────────
{
  "data": [
    { "id": 1, "name": "Alice" },
    { "id": 2, "name": "Bob" }
  ],
  "meta": {
    "total": 200,
    "page": 1,
    "per_page": 20,
    "next": "/users?page=2"
  }
}

// ── Error response (RFC 7807 Problem Details) ─────────────────
{
  "type":   "https://example.com/errors/validation",
  "title":  "Validation Failed",
  "status": 422,
  "detail": "The 'email' field is not a valid address.",
  "errors": [
    { "field": "email", "message": "Invalid format" }
  ]
}

Notes

"data" envelope — wrapping the payload lets you add metadata later without breaking clients
"meta" block — pagination, counts, and cursor values live here, not inside data
"next" cursor — cursor-based pagination is safer than page numbers for live data
RFC 7807 error shape — type, title, status, detail are the standard fields
"errors" array — per-field validation errors alongside the top-level error message
Always return JSON error bodies even on 4xx/5xx — never let plain-text stack traces leak

Common Mistakes

  • Returning a 200 status with an error message in the body — use the correct 4xx/5xx code
  • Forgetting Content-Type: application/json on POST/PUT — server may reject or misparse the body
  • Trying to parse a 204 No Content response body — it is empty by definition
  • Exposing stack traces or internal error messages in 500 responses

Tips

  • Always check the HTTP status code before parsing the body — don't assume success
  • Adopt RFC 7807 Problem Details for error responses — it's becoming the industry standard
  • Use cursor-based pagination ("next_cursor") instead of page numbers for stable, live data sets

Summary

  • Set Content-Type: application/json on every request with a body
  • Wrap responses in a data envelope to allow adding meta later
  • Use RFC 7807 for consistent, machine-readable error responses
  • Check HTTP status code first — never trust a 200 blindly
