The Six JSON Types

JSON defines exactly six value types. There is no distinction between integers and floats — all numbers share a single type. There are no date, undefined, function, or binary types; these must be represented as strings or omitted. Understanding the limits of each type prevents silent data loss, especially with large integers and special float values like Infinity and NaN, which are not valid JSON.

Type Quick Rules

  • string — always double-quoted; escape special chars with \
  • number — no NaN, no Infinity, no leading zeros (except 0.x)
  • boolean — only true or false (lowercase)
  • null — represents absence of a value; distinct from false or 0
  • object — unordered collection of key/value pairs wrapped in { }
  • array — ordered list of values wrapped in [ ]; may mix types
  • JSON has no undefined — missing keys and null values are different things

All Six Types in One Document

{
  "type_string":  "Hello, World!",
  "type_escape":  "Line 1\nLine 2\t(tab)",
  "type_unicode": "\u00e9l\u00e8ve",

  "type_integer": 42,
  "type_float":   3.14159,
  "type_exp":     2.5e10,
  "type_neg":     -7,

  "type_true":    true,
  "type_false":   false,

  "type_null":    null,

  "type_array":   [1, "two", true, null],

  "type_object":  { "x": 1, "y": 2 }
}

Notes

\n, \t — escape sequences inside strings; raw newlines not allowed
\uXXXX — any Unicode character by hex code point
2.5e10 — scientific notation is valid; equals 25000000000
-7 — negative numbers use a leading minus; no leading plus allowed
Arrays may contain mixed types — no schema enforcement in bare JSON
null ≠ false ≠ 0 ≠ "" — each is a distinct value

Common Mistakes

  • Using NaN or Infinity in number fields — neither is valid JSON
  • Storing dates as numbers (Unix timestamps) without documenting the unit (ms vs s)
  • Assuming large integers are safe — parsers using IEEE 754 doubles lose precision above 2⁵³
  • Treating null and a missing key as equivalent — they have different semantics

Tips

  • Represent dates as ISO 8601 strings: "2024-06-15T14:30:00Z"
  • For large integers (IDs, snowflakes), use strings to avoid precision loss across languages
  • Use null explicitly to distinguish "value is absent" from "value is empty string"

Summary

  • Six types: string · number · boolean · null · object · array
  • Numbers: no NaN, no Infinity, no leading zeros (except 0.x)
  • Strings must be double-quoted; use \uXXXX for Unicode escapes
  • Arrays are ordered and may mix types; objects are unordered key-value maps
