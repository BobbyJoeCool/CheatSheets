Objects & Arrays

A JSON object is an unordered collection of zero or more key/value pairs enclosed in curly braces { }. Keys must be unique strings within one object (duplicate keys are technically undefined behaviour). A JSON array is an ordered sequence of zero or more values enclosed in square brackets [ ]. Arrays preserve insertion order and may contain any mix of the six JSON types, including other arrays or objects, enabling arbitrarily deep nesting.

Key Rules

  • Object key order is not guaranteed — never rely on it for logic
  • Duplicate keys in one object: spec says undefined; most parsers keep the last value
  • Array elements are accessed by zero-based index, not by key
  • Empty object {} and empty array [] are both valid JSON values
  • Arrays and objects can nest to any depth — but keep it manageable for humans
  • A single comma separates each element; no comma after the last

Objects, Arrays & Nesting

{
  "user": {
    "id": 1,
    "name": "Bob",
    "roles": ["admin", "editor"],
    "address": {
      "city": "Denver",
      "coords": [39.7392, -104.9903]
    }
  },

  "empty_obj": {},
  "empty_arr": [],

  "matrix": [
    [1, 2, 3],
    [4, 5, 6]
  ],

  "mixed": [42, "hello", true, null, {"k": "v"}]
}

Notes

"roles": [...] — array as an object value; common pattern for tags/permissions
"coords": [lat, lng] — array of numbers; positional meaning is by convention
"empty_obj": {} — valid; represents an object with no properties yet
"empty_arr": [] — valid; represents a list with zero items
"matrix" — array of arrays; creates a 2D structure
"mixed" — arrays can contain any type mix including nested objects

PatternExampleNotes

Common Mistakes

  • Relying on object key order — it is not guaranteed by the spec or all parsers
  • Using duplicate keys in the same object — behaviour is undefined and parser-dependent
  • Trailing comma after last array element or object property
  • Confusing positional array meaning — [lat, lng] vs [lng, lat] requires documentation

Tips

  • Prefer arrays of objects with named keys over positional arrays for clarity
  • Use empty arrays [] rather than null for absent list values — easier to iterate
  • Keep nesting to 3–4 levels maximum; deeper structures are a sign to redesign the schema

Summary

  • Objects: unordered key/value pairs in { } — keys are unique strings
  • Arrays: ordered values in [ ] — zero-indexed, mixed types allowed
  • Both can be empty ({}, []) and nested to any depth
  • Array of objects is the most common shape for list responses in REST APIs
