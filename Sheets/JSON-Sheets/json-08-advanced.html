Advanced & Related Formats

JSON5 · NDJSON · BSON · JWT · GeoJSON · config patterns

The JSON Ecosystem

JSON's strict syntax makes it reliable but sometimes inconvenient — no comments, no trailing commas, no binary data. The ecosystem has produced several supersets and relatives that solve specific pain points. JSON5 adds comments and unquoted keys for config files. NDJSON enables streaming of record sequences. BSON adds binary types for database storage. JWT uses Base64-encoded JSON for authentication tokens. Understanding which format to reach for — and why — is as important as knowing JSON itself.

Format Selection Guide

  • Config files with comments → JSON5 or JSONC
  • Streaming / log lines → NDJSON / JSON Lines
  • Database storage with binary data → BSON (MongoDB)
  • Auth tokens in HTTP headers → JWT
  • Geographic features → GeoJSON (RFC 7946)
  • REST API payloads → plain JSON (RFC 8259)
  • High-performance binary APIs → MessagePack or CBOR

Format Examples

// ── JSON5 (config file) ───────────────────────────────────────
{
  // Comments are allowed
  name: 'Alice',            // unquoted keys
  score: +9.5,              // leading + allowed
  hex: 0xFF,                // hex literals
  multiline: "line 1\
line 2",                    // backslash line continuation
  trailing: "ok",           // trailing comma OK
}

// ── NDJSON / JSON Lines (one object per line) ─────────────────
{"id":1,"event":"login","ts":"2024-01-01T00:00:00Z"}
{"id":2,"event":"view","ts":"2024-01-01T00:01:00Z"}
{"id":3,"event":"logout","ts":"2024-01-01T00:05:00Z"}

// ── JWT structure (header.payload.signature) ──────────────────
// Header  (Base64url-decoded):
{"alg":"HS256","typ":"JWT"}
// Payload (Base64url-decoded):
{"sub":"1234567890","name":"Alice","iat":1516239022}
// Full token:
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxMjM0In0.SflKxwR...

// ── GeoJSON (RFC 7946) ────────────────────────────────────────
{
  "type": "Feature",
  "geometry": {
    "type": "Point",
    "coordinates": [-104.9903, 39.7392]
  },
  "properties": { "name": "Denver" }
}

Notes

JSON5 is a superset — all valid JSON is valid JSON5, but not vice versa
NDJSON: newline-delimited — stream parsers read one line at a time, no full-document parse needed
JWT payload is Base64url encoded, not encrypted — never put secrets in it without encryption
GeoJSON coordinates are always [longitude, latitude] — longitude first (common gotcha)
BSON adds types: Date, ObjectId, Binary, Decimal128 — only used with BSON-aware drivers (MongoDB)
JSONC (JSON with Comments) is used by VS Code settings.json and TypeScript tsconfig.json

Common Mistakes

  • Treating JWT as encrypted — the payload is only Base64-encoded and is publicly readable
  • GeoJSON: using [lat, lng] order — the spec requires [longitude, latitude]
  • Feeding NDJSON to a standard JSON parser — it will fail; parse line-by-line instead
  • Using JSON5 in a production API — most HTTP clients and servers expect strict JSON

Tips

  • Use NDJSON for log ingestion pipelines — append a line per event, no file rewriting needed
  • Inspect JWT payloads at jwt.io — paste the token to decode and verify
  • For config files that developers edit directly, JSON5 or JSONC is friendlier than strict JSON

Summary

  • JSON5 / JSONC add comments & trailing commas — good for config, not APIs
  • NDJSON: one JSON object per line — built for streaming and log pipelines
  • JWT: Base64url-encoded JSON header + payload — not encrypted by default
  • GeoJSON: standard geographic format — coordinates are always [lng, lat]
