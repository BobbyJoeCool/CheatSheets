JSON Schema

type validation · required · properties · $ref · $defs · allOf · anyOf · oneOf

What Is JSON Schema?

JSON Schema is a vocabulary (draft standardized at json-schema.org) for describing the structure and constraints of JSON documents. A schema is itself a JSON object containing keywords like type, properties, and required. Validators (ajv, jsonschema, etc.) use the schema to check whether a JSON instance conforms. Schemas are used for API documentation, form validation, code generation, and config file enforcement. The current stable release is Draft 2020-12.

Core Keywords

  • $schema — declares which draft is being used
  • type — constrains to one or more JSON types
  • properties — defines allowed keys and their sub-schemas for objects
  • required — array of keys that must be present
  • $defs — named reusable sub-schemas (replaces old definitions)
  • $ref — reference to another schema by JSON Pointer URI
  • additionalProperties — false to disallow unlisted keys

Full Schema Example

{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://example.com/user.schema.json",
  "title": "User",
  "type": "object",
  "required": ["id", "name", "email"],
  "properties": {
    "id":    { "type": "integer", "minimum": 1 },
    "name":  { "type": "string",  "minLength": 1, "maxLength": 100 },
    "email": { "type": "string",  "format": "email" },
    "age":   { "type": "integer", "minimum": 0, "maximum": 150 },
    "role":  { "$ref": "#/$defs/role" },
    "tags":  { "type": "array", "items": { "type": "string" } }
  },
  "additionalProperties": false,
  "$defs": {
    "role": {
      "type": "string",
      "enum": ["admin", "editor", "viewer"]
    }
  }
}

Notes

$schema — tells validators which draft rules to apply
$id — canonical URI for this schema; enables cross-schema $ref
"required": [...] — keys listed here must exist in the instance
"minimum": 1 — number constraint; see table for all numeric keywords
"format": "email" — semantic hint; validators may or may not enforce it
"$ref": "#/$defs/role" — reuses the role definition inline
"additionalProperties": false — rejects any key not listed in properties

Common Mistakes

  • Confusing anyOf and oneOf — use oneOf only when exactly one branch should match
  • Omitting additionalProperties: false then being surprised extra keys are accepted
  • Using old definitions key — it was replaced by $defs in Draft 2019-09
  • Assuming format is enforced — most validators treat it as advisory by default

Tips

  • Use $defs to DRY up repeated sub-schemas and reference with $ref
  • Set "additionalProperties": false on API payloads to catch typos in field names
  • Test schemas with jsonschema.dev or ajv-cli before deploying

Summary

  • JSON Schema validates JSON structure using keyword-based rules
  • type · properties · required are the three most-used keywords
  • $defs + $ref enable reusable, composable schemas
  • allOf / anyOf / oneOf combine schemas with AND / OR / XOR logic
