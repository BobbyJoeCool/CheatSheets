CSS · Sheet 1

Architecture & Organization

semantic grouping · file architecture · cascade order · specificity

Description

Well-organized CSS is written in a consistent order, grouped by concern, and split into purposeful layers. The four semantic groups — Positioning, Box Model, Typography, and Visuals — give every property a logical home inside every rule you write. Layering files as Base → Layout → Components → Utilities mirrors the natural cascade, keeping specificity low and overrides predictable. Consistent structure across a project means any developer can find, change, or debug a rule in seconds.

Notes

  • Property order within a rule is a convention, not enforced by the browser — but enforceable with stylelint-order
  • The cascade reads files top-to-bottom; later rules with equal specificity win
  • !important breaks the cascade — avoid in components, reserve for utility overrides only
  • CSS @layer lets you declare layer order explicitly, making the architecture enforceable in-browser

Cascade Order & @layer

Baseresets · tokens
Layoutstructure · grid
ComponentsUI elements
Utilitiesoverrides · helpers
/* main.css — declare layer order first, then import each file into its layer */
@layer base, layout, components, utilities;

@import "base/_reset.css"        layer(base);
@import "base/_variables.css"    layer(base);
@import "layout/_grid.css"       layer(layout);
@import "components/_button.css" layer(components);
@import "utilities/_spacing.css" layer(utilities);

⚠ Common Mistakes

  • Writing properties in random order — makes rules hard to scan and diff
  • Using #id selectors in components — locks in high specificity, hard to override
  • Putting layout rules inside component files — breaks separation of concerns
  • Overusing !important to fix specificity wars instead of restructuring

✓ Tips & Best Practices

  • Use a linter (stylelint + stylelint-order) to enforce property order automatically
  • Keep specificity flat — prefer classes over chained element+class selectors
  • Use @layer in new projects — makes cascade order explicit and debuggable
  • One component = one file; name the file after the component (_card.css)

Quick Summary

  • Group properties: Pos → Box → Type → Visual
  • Layer files: Base → Layout → Components → Utilities
  • Later rules win ties; higher specificity always wins
  • @layer lets you control cascade order explicitly
