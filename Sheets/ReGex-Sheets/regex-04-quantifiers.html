Sheet 04

Quantifiers

* + ? {n,m} · greedy · lazy · possessive · repetition

How Quantifiers Work

A quantifier follows a token (character, class, or group) and specifies how many times that token must appear for a match. By default, quantifiers are greedy — the engine matches as many characters as possible, then backs off only as needed to satisfy the rest of the pattern. Adding ? after a quantifier makes it lazy (match as few as possible). Some engines support possessive quantifiers (++, *+) that never backtrack.

Key Rules

  • Quantifiers apply to the immediately preceding token or group.
  • * allows zero matches — it never causes a match to fail on its own.
  • Greedy vs lazy only matters when the pattern has content after the quantifier.
  • {n} requires exactly n repetitions; {n,} means n or more.
  • Possessive quantifiers (*+, ++) are not supported in JavaScript or Python's re module (use regex package).

Greedy vs Lazy Comparison

PatternInputMatch
<.+>"<a><b>""<a><b>" (greedy)
<.+?>"<a><b>""<a>" (lazy)
".*""x" and "y""x" and "y" (greedy — full span)
".*?""x" and "y""x" (lazy — stops at first ")
\d{2,4}"12345""1234" (greedy — takes 4)
\d{2,4}?"12345""12" (lazy — takes 2)

Quantifier Examples

// ── Greedy (default) ──────────────────────────────────────────
'bold'.match(/<.+>/)     // ["bold"] — eats everything
'aaa'.match(/a*/)               // ["aaa"]
'bbb'.match(/a*/)               // [""] — zero match succeeds!

// ── Lazy ──────────────────────────────────────────────────────
'bold'.match(/<.+?>/)    // [""]  — stops at first >
'bold'.match(/<.+?>/g)   // ["", ""]

// ── Exact & range ─────────────────────────────────────────────
/\d{4}/.test('2024')            // true
/\d{4}/.test('202')             // false
/^\d{3}-\d{4}$/.test('555-1234') // true  — phone segment
/\b\w{5,}\b/g                   // all words 5+ chars

// ── Optional with ? ───────────────────────────────────────────
/https?:\/\//.test('http://')   // true
/https?:\/\//.test('https://')  // true
/Jan(?:uary)?/.test('January')  // true
/Jan(?:uary)?/.test('Jan')      // true

// ── Avoiding catastrophic backtracking ───────────────────────
// BAD:  /^(a+)+$/.test('aaaaaab')  — exponential backtrack
// GOOD: /^a+$/.test('aaaaaab')     — no nested quantifier

Notes

.+ greedy — expands to end of string, then backtracks to find >
.+? lazy — tries shortest match first, expands only if needed
a* — matches empty string; can match at every position
\d{3}-\d{4} — fixed-count quantifiers are ideal for structured data like phone numbers
s? — makes the preceding character optional; common for singular/plural
Nested quantifiers like (a+)+ cause catastrophic backtracking — keep quantifier nesting flat

Common Mistakes

  • Using .* when .+ is intended — * succeeds on an empty string, masking logic errors.
  • Forgetting that ? after a quantifier means lazy, not "optional" — a+? ≠ a?.
  • Nesting quantifiers: (a+)+ causes catastrophic backtracking on non-matching input.
  • Writing {3,5} with a space: {3, 5} — most engines treat the space as a literal, breaking the quantifier.

Tips

  • Prefer lazy quantifiers when matching delimited content like HTML tags or quoted strings.
  • Use exact counts ({n}) for structured formats (dates, IDs, codes) — it's self-documenting.
  • Watch for .* in anchored patterns — if the string can be empty, this always matches.

Summary

  • * / + / ? — zero-or-more, one-or-more, zero-or-one (all greedy by default).
  • {n,m} — explicit repetition bounds for precise control.
  • Add ? after any quantifier to make it lazy (shortest match).
  • Avoid nested quantifiers to prevent catastrophic backtracking.
