Regex Across Languages

The pattern syntax is nearly identical across all major languages — the same \d+, (?:…), and (?<name>…) work everywhere. What differs is the API: how you compile, execute, and retrieve results. Understanding these differences lets you translate a pattern from one language to another with minimal effort. The table below maps the same five operations across Python, JavaScript, Java, PHP, and Go.

Key Differences

  • Python uses raw strings (r"…") to avoid double-escaping backslashes.
  • JavaScript regex literals (/pattern/flags) don't need quotes; strings passed to new RegExp() do require double-escaping.
  • Java requires Pattern.compile() and double-escaped strings: "\\d+".
  • Go's regexp package uses RE2 syntax — no lookaheads or backreferences.
  • PHP uses PCRE via preg_* functions; the pattern must include delimiters: '/pattern/flags'.

Side-by-Side: Extract Year from Date String

# ── Python ────────────────────────────────────────────────────
import re
m = re.search(r'(?P\d{4})-\d{2}-\d{2}', '2024-01-15')
m.group('year')           # '2024'
m.group(1)                # '2024'

# ── JavaScript ────────────────────────────────────────────────
const m = '2024-01-15'.match(/(?\d{4})-\d{2}-\d{2}/);
m.groups.year;            // '2024'
m[1];                     // '2024'

# ── Java ──────────────────────────────────────────────────────
Pattern p = Pattern.compile("(?\\d{4})-\\d{2}-\\d{2}");
Matcher m = p.matcher("2024-01-15");
if (m.find()) m.group("year");  // "2024"

# ── PHP ───────────────────────────────────────────────────────
preg_match('/(?P\d{4})-\d{2}-\d{2}/', '2024-01-15', $m);
$m['year'];               // "2024"

# ── Go (no named groups — use index) ─────────────────────────
re := regexp.MustCompile(`(\d{4})-\d{2}-\d{2}`)
m  := re.FindStringSubmatch("2024-01-15")
m[1]                      // "2024"

Notes

Python — (?P<name>…) is the named group syntax; access via .group('name')
JavaScript — (?<name>…) syntax; result in match.groups.name (ES2018+)
Java — double-escape all backslashes in string literals: "\\d+" not "\d+"
PHP — wrap pattern in delimiters: '/pattern/flags'; named groups use (?P<name>…)
Go (RE2) — no lookaheads, no backreferences, no named groups; use submatch indices instead
Go inline flags — (?i) at start of pattern enables case-insensitive matching without a flag argument

Common Mistakes

  • Forgetting to double-escape in Java: "\d+" is a string escape error; always write "\\d+".
  • Omitting delimiters in PHP: preg_match('\d+', …) causes a warning — always use '/\d+/'.
  • Using lookaheads or backreferences in Go — RE2 intentionally omits them; restructure your pattern or switch to a PCRE library.
  • Using re.match() in Python thinking it searches the whole string — it only checks from the start.

Tips

  • Compile patterns once and reuse: re.compile() in Python, Pattern.compile() in Java, regexp.MustCompile() in Go.
  • In Go, use raw string literals (`…`) to avoid double-escaping — the same benefit as Python's r"…".
  • When porting a PCRE pattern to Go, check for lookaheads and backreferences first — those are the most common blockers.

Summary

  • Pattern syntax is universal — API wrappers are what differ across languages.
  • Python: raw strings · re.search/findall/sub · named groups via (?P<name>…).
  • Java: double-escape strings · compile once · Matcher.find() loop for all matches.
  • Go (RE2): no lookaheads or backreferences · use raw backtick strings · submatch index arrays.
