Flowcharts

Node shapes · connectors · subgraphs · direction · edge labels · styling

What is a Flowchart?

A flowchart maps a process as a sequence of steps and decisions using nodes (shapes) connected by arrows. In Mermaid, flowchart is the most versatile and widely-used diagram type. Direction is set on the first line: TD (top→down), LR (left→right), BT (bottom→top), RL (right→left). Typical uses: software logic, user journeys, decision trees, CI/CD pipelines, onboarding flows.

Use Cases

  • Software / algorithm logic
  • User journey & UX flows
  • Business process documentation
  • Decision trees & if/else maps
  • CI/CD pipeline visualization
  • Troubleshooting / runbooks

Code Example — User Authentication Flow

%%{init: {"theme": "dark"}}%%
flowchart TD
  A([User visits /login]) --> B{Session\nvalid?}
  B -- Yes --> C[/Redirect to dashboard/]
  B -- No  --> D[Show login form]
  D --> E[/User submits form/]
  E --> F{Credentials\ncorrect?}
  F -- Yes --> G[(Save session\nto DB)]
  F -- No  --> H[Show error]
  G --> C
  H --> D
  subgraph auth [Auth Service]
    F
    G
  end
  style H fill:#7b2020,color:#ffcccc
  style C fill:#1a4d2e,color:#b9f0cb

Node Shapes

Syntax Shape
id[Text] Rectangle — default process step
id(Text) Rounded rectangle
id([Text]) Stadium / pill — start or end
id{Text} Diamond — decision / branch
id{{Text}} Hexagon
id[/Text/] Parallelogram — input / output
id[\Text\] Parallelogram (alt direction)
id[/Text\] Trapezoid
id[\Text/] Trapezoid (inverted)
id[(Text)] Cylinder — database
id((Text)) Circle
id(((Text))) Double circle
id>Text] Asymmetric / flag shape

Connectors & Arrows

Syntax Meaning
A --> B Solid arrow (open head)
A --- B Line — no arrowhead
A -- label --> B Arrow with edge label
A -->|label| B Arrow with label (alt syntax)
A ==> B Thick arrow
A == label ==> B Thick arrow with label
A -.-> B Dotted arrow
A -. label .-> B Dotted arrow with label
A --o B Circle end
A --x B Cross / terminated end
A <--> B Bidirectional arrow
A o--o B Circle on both ends
A x--x B Cross on both ends

Subgraphs & Styling

Syntax Purpose
subgraph id [Title] Open a named subgraph group
end Close subgraph block
direction LR Override direction inside a subgraph
style id fill:,color: Inline style a single node
classDef name fill:,stroke: Define a reusable CSS class
class A,B name Apply a classDef to listed nodes
A:::name --> B Apply class inline on a node
linkStyle 0 stroke:#f00 Style an edge by 0-based index
\n in label Line break inside a node label
"quoted id" Node ID containing spaces

Common Mistakes

  • Using &, <, or > in labels without quoting — wrap label in "…"
  • Forgetting end to close a subgraph — diagram silently fails to render
  • Re-using the same node ID with different labels — IDs are global; reuse merges nodes
  • linkStyle index is 0-based and counts every edge in declaration order

Tips & Best Practices

  • Prefer LR for process flows; use TD for hierarchies and trees
  • Use classDef + class instead of repeating style on every node
  • Group related nodes in subgraphs — makes large diagrams scannable
  • Use stadium ([…]) for start/end; reserve diamond {…} for decisions only

Summary

  • Declare with flowchart TD|LR|BT|RL — direction keyword is required
  • 13 node shapes; shape is set by bracket type, not a keyword
  • 13 connector styles: solid, dotted, thick, labeled, bidirectional
  • Subgraphs group nodes visually; classDef enables reusable styling
