HTML · Sheet 14

Accessibility

ARIA roles · states · properties · keyboard · focus · live regions

Description

ARIA (Accessible Rich Internet Applications) is a set of attributes that supplement HTML semantics when native elements aren't sufficient — primarily for custom widgets. The golden rule: use semantic HTML first. A native <button> beats a <div role="button"> every time — it's keyboard accessible, correctly announced, and needs no ARIA. Use ARIA only to fill the gaps: communicating dynamic state changes, labelling regions without visible headings, and announcing live updates. ARIA never makes an element keyboard-accessible — you must add that with tabindex and event listeners yourself.

Notes

  • Rule of ARIA: no ARIA is better than bad ARIA — incorrect roles actively mislead screen reader users
  • aria-hidden="true" hides from screen readers but the element remains visible and focusable — never hide interactive elements with it
  • tabindex="0" adds to natural tab order; tabindex="-1" allows programmatic focus only; positive values (avoid) override natural order
  • aria-live="polite" waits for the user to finish before announcing; assertive interrupts immediately — use assertive sparingly
  • WCAG 2.1 AA is the most widely required standard — the four principles are Perceivable, Operable, Understandable, Robust (POUR)

Common ARIA Roles

role= Use when… Native equivalent
button Custom clickable div/span <button>
checkbox Custom toggle control <input type="checkbox">
dialog Custom modal overlay <dialog>
alert Error or important message No native equivalent
status Non-urgent live update No native equivalent
tab Tab control in a tablist No native equivalent
tabpanel Tab panel content area No native equivalent
tooltip Hover/focus description No native equivalent
progressbar Progress indicator <progress>

Key ARIA Attributes

Attribute Purpose
aria-label Provides a name when no visible label exists
aria-labelledby Points to id(s) of visible labelling elements
aria-describedby Points to id(s) of descriptive/error elements
aria-hidden Hides decorative elements from screen readers
aria-expanded true/false — open/closed state of menus, accordions
aria-checked true/false/mixed — checked state of custom checkboxes
aria-selected true/false — selected state in listbox, tabs
aria-disabled true — communicates disabled without removing focusability
aria-invalid true — marks a field as having a validation error
aria-live polite / assertive — region announces dynamic changes
aria-atomic true — read the entire region on any change

♿ Five Rules of ARIA Use

  • 1. Use native HTML elements and attributes instead of ARIA whenever possible
  • 2. Don't change native semantics unless you really have to
  • 3. All interactive ARIA controls must be keyboard usable
  • 4. Don't use role="presentation" or aria-hidden="true" on focusable elements
  • 5. All interactive elements must have an accessible name

Example

<!-- ① Native HTML beats ARIA — prefer semantic elements -->
<!-- WRONG: -->
<!-- <div role="button" tabindex="0"
          onclick="save()" onkeydown="handleKey(event)">Save</div> -->
<!-- CORRECT: -->
<button type="button" onclick="save()">Save</button>
<!-- ② aria-label: accessible name when no visible label exists -->
<button aria-label="Close dialog">
  <svg aria-hidden="true" focusable="false"><!-- icon --></svg>
</button>
<!-- ③ aria-labelledby: point to a visible element as the label -->
<section aria-labelledby="pricing-heading">
  <h2 id="pricing-heading">Pricing</h2>
  <!-- section content -->
</section>
<!-- ④ aria-describedby + aria-invalid: field with error -->
<label for="email">Email</label>
<input type="email" id="email"
       aria-describedby="email-error"
       aria-invalid="true" />
<p id="email-error" role="alert">Please enter a valid email address.</p>
<!-- ⑤ aria-expanded: toggle button controlling a panel -->
<button aria-expanded="false" aria-controls="faq-answer" id="faq-q">
  What is your return policy?
</button>
<div id="faq-answer" role="region" aria-labelledby="faq-q" hidden>
  <!-- answer — JS toggles hidden + aria-expanded together -->
</div>
<!-- ⑥ Tab pattern: tablist > tab + tabpanel -->
<div role="tablist" aria-label="Product views">
  <button role="tab" aria-selected="true"
          aria-controls="panel-details" id="tab-details">Details</button>
  <button role="tab" aria-selected="false"
          aria-controls="panel-reviews" id="tab-reviews"
          tabindex="-1">Reviews</button>
</div>
<div role="tabpanel" id="panel-details" aria-labelledby="tab-details">
  <!-- details content -->
</div>
<div role="tabpanel" id="panel-reviews" aria-labelledby="tab-reviews" hidden>
  <!-- reviews content -->
</div>
<!-- ⑦ aria-live: announce dynamic content without a page reload -->
<div aria-live="polite" aria-atomic="true">
  <p id="status-msg"><!-- JS writes "3 results found" here --></p>
</div>

<!-- role="alert" is shorthand for aria-live="assertive" —
     use only for urgent, time-sensitive messages -->
<div role="alert">
  Session expiring in 2 minutes. Save your work.
</div>
<!-- ⑧ Skip link — first focusable element, visually hidden until focused -->
<a href="#main-content" class="skip-link">Skip to main content</a>
<!-- CSS: .skip-link { position:absolute; left:-9999px; }
          .skip-link:focus { left:0; }                    -->
		  <!-- ⑨ Focus management: move focus after dialog opens -->
<dialog>
  <!-- tabindex="-1" allows programmatic focus on a non-interactive element -->
  <h2 tabindex="-1">Confirm Action</h2>
  <!-- JS: dialog.showModal(); dialog.querySelector('h2').focus(); -->
  <button>Cancel</button>
  <button>Confirm</button>
</dialog>

⚠ Common Mistakes

  • Adding role="button" to a <div> without also adding tabindex="0" and keyboard handlers — role alone doesn't make it keyboard accessible
  • Using aria-hidden="true" on a focusable element — screen reader users can still tab to it but get no announcement
  • Using aria-live="assertive" for all dynamic updates — it interrupts users mid-sentence; reserve for critical, time-sensitive errors
  • Updating aria-expanded without showing/hiding the controlled content — the state and the visual must match
  • Using aria-label to override a perfectly good visible label — this creates a mismatch between what's spoken and what's seen

✓ Tips & Best Practices

  • Test with an actual screen reader (NVDA + Chrome on Windows, VoiceOver + Safari on Mac) — the Accessibility Tree alone isn't enough
  • Navigate your page with keyboard only (Tab, Shift+Tab, Enter, Space, Arrow keys) — this catches missing focus, wrong order, and focus traps
  • Use the ARIA Authoring Practices Guide (APG) at w3.org/WAI/ARIA/apg for widget patterns — keyboard expectations are documented per role
  • Add :focus-visible styles — removes the focus ring on mouse click while keeping it for keyboard users

Quick Summary

  • Semantic HTML first — ARIA supplements, never replaces, native elements
  • ARIA provides role, name, and state — but never keyboard behavior; add that separately
  • aria-live="polite" for status updates; role="alert" for urgent errors
  • Test with keyboard + screen reader — automated tools catch ~30% of a11y issues at most
