Generic Containers

div · span · block vs inline · data-* attributes

Description

<div> and <span> are the workhorses of HTML layout — generic containers with no semantic meaning and no default styling beyond their display type. <div> is block-level (starts a new line, full available width) and is used for layout wrappers, card shells, and CSS/JS hooks. <span> is inline (flows within text) and is used to target a portion of text for styling or scripting. Use them when no semantic element fits — always prefer a semantic element first.

Notes

  • <div> and <span> carry zero semantic meaning — they provide no role to screen readers
  • Block elements stack vertically; inline elements flow horizontally within a line
  • data-* attributes let you store custom data on any element — accessed in JS via element.dataset
  • CSS can change any element's display type (display: block on a <span>) — but this doesn't change its semantic meaning
  • Nesting block elements inside inline elements is invalid HTML — e.g. <span><div>…</div></span> is wrong

Block vs Inline Elements

Property Block Inline
Default display Starts on new line Flows in text
Default width Full parent width Fits content
Can contain Block + inline Inline only
Examples <div> <p> <h1> <span> <a> <strong>
Generic element <div> <span>

data-* Attribute API

HTML attribute JS access via dataset
data-user-id="42" el.dataset.userId → "42"
data-theme="dark" el.dataset.theme → "dark"
data-max-count="10" el.dataset.maxCount → "10"
CSS targeting [data-theme="dark"] { … }

♿ Accessibility

  • <div> and <span> are invisible to screen readers as structural elements — add role + aria-label only if making them interactive
  • Never use a <div> with a click handler as a button — use <button>; it gets keyboard support and role for free
  • data-* attributes are not exposed to assistive technology — don't use them to convey meaningful information; use aria-* or semantic HTML instead
  • If you must use <div role="button">, also add tabindex="0" and keyboard event handlers — a significant amount of work that <button> handles automatically

Example

<!-- ① div: block-level layout wrapper — no semantic meaning -->
<div class="card">
  <div class="card__header">
    <h2><!-- card title --></h2>
  </div>
  <div class="card__body">
    <!-- card body content -->
  </div>
</div>

<!-- ② span: inline wrapper — targets a portion of text -->
<p>The price is <span class="price">$19.99</span> per month.</p>

<!-- ③ data-* attributes: store custom data on any element -->
<button
  data-action="add-to-cart"
  data-product-id="sku-447"
  data-price="19.99">
  Add to Cart
</button>
<!-- JS access via dataset API:
  btn.dataset.action      → "add-to-cart"
  btn.dataset.productId   → "sku-447"   (hyphen becomes camelCase)
  btn.dataset.price       → "19.99"
-->

<!-- ④ data-* for CSS state targeting (no JS required) -->
<div data-theme="dark">
  <!-- content inheriting dark theme styles -->
</div>
<!-- CSS: [data-theme="dark"] { background: #111; color: #eee; } -->

<!-- ⑤ Common div patterns -->

<!-- Layout wrapper -->
<div class="grid">
  <div class="grid__item"><!-- item content --></div>
  <div class="grid__item"><!-- item content --></div>
</div>

<!-- JS mount point -->
<div id="map-container"></div>

<!-- ⑥ INVALID — block element nested inside inline element -->
<!-- <span><div>Wrong!</div></span> ← browser repairs it badly -->

<!-- ⑦ WRONG — div as a fake button -->
<!-- <div onclick="save()">Save</div> ← no keyboard, no role -->
<!-- CORRECT: -->
<button type="button" onclick="save()">Save</button>

⚠ Common Mistakes

  • "Div soup" — nesting <div>s six layers deep when semantic elements would do the job
  • Using <div> for interactive elements (clickable cards, fake buttons) without role or keyboard support
  • Nesting a block element (<div>, <p>) inside an inline element (<span>, <a>)
  • Using data-* to store info that should be in aria-* — screen readers don't read data-*
  • Relying on CSS display: block on a <span> instead of using a <div> — confusing and fragile

✓ Tips & Best Practices

  • Reach for a semantic element first — only use <div>/<span> when nothing else fits
  • Use data-* for JS configuration and CSS state hooks — they're stable across refactors
  • Give layout <div>s class names that describe their role (card, grid, modal-overlay)
  • Use id sparingly on <div>s — prefer class for styling, data-* for JS hooks

Quick Summary

  • <div> = block generic container; <span> = inline generic container
  • Neither carries semantic meaning — always prefer a semantic element
  • data-* stores custom data; accessed in JS via element.dataset.camelCase
  • Never put block elements inside inline elements — it's invalid HTML
