HTML · Sheet 1

Architecture

document structure · elements · attributes · entities · parsing

Description

HTML (HyperText Markup Language) is the skeleton of every web page — a text file of elements that tell the browser what content exists and what it means, not how it looks. A browser reads the HTML file top-to-bottom, builds a tree of objects called the DOM (Document Object Model), and renders that tree to the screen. CSS handles appearance; JavaScript handles behavior; HTML handles structure and meaning.

Notes

  • HTML is not a programming language — it has no logic, loops, or variables
  • Browsers never throw errors on bad HTML — they silently repair and continue
  • File extension is .html (or .htm — legacy, avoid)
  • HTML is case-insensitive, but lowercase tags and attributes are the standard
  • The current standard is HTML5 (living standard, maintained by WHATWG)

Element Types

Type Description
Container Has opening + closing tag; wraps content — <p>, <div>, <a>
Void No closing tag; no children — <img>, <br>, <input>, <meta>
Block Starts a new line, full width — <p>, <div>, <h1>
Inline Flows within text — <span>, <a>, <strong>
Comment Ignored by browser — <!-- text -->

Common Character Entities

Entity Char When to use
&lt; < Less-than sign in text
&gt; > Greater-than sign in text
&amp; & Ampersand in text or attr
&quot; " Quote inside an attribute
&nbsp; ␣ Non-breaking space
&copy; © Copyright symbol
&mdash; — Em dash

Architectural Tags

Tag Description
<!DOCTYPE html> Declares HTML5; must be the very first line — no content before it
<html> Root element; lang attribute is required for a11y
<head> Metadata container — nothing here renders visibly on the page
<body> All visible page content goes here; only one allowed per document
<title> Browser tab text; required — affects SEO and a11y
<meta> Machine-readable metadata: charset, viewport, description
<link> Links external resources (stylesheets, icons) into the document
<script> Embeds or links JavaScript; use defer to avoid blocking parse

♿ Accessibility

  • Semantic elements give screen readers free role, name, and state — always prefer them over <div>
  • The lang attribute on <html> tells screen readers which language to pronounce content in — always include it
  • Document reading order follows DOM order — structure your HTML in the order a screen reader should encounter it
  • A well-structured HTML document is the single highest-impact a11y investment

Example

<!-- ① DOCTYPE: tells the browser this is modern HTML5 -->
<!DOCTYPE html>

<!-- ② Root element: lang sets the document language -->
<html lang="en">

  <!-- ③ Head: metadata — nothing here renders visibly -->
  <head>
    <meta charset="UTF-8" />          <!-- character encoding -->
    <meta name="viewport"
          content="width=device-width,
                   initial-scale=1.0" /> <!-- mobile scaling -->
    <title>Page Title</title>          <!-- browser tab text -->
  </head>

  <!-- ④ Body: everything visible goes here -->
  <body>

    <!-- Container element: opening tag + content + closing tag -->
    <p class="intro" id="lead">Hello, world.</p>
    <!--    ↑ attribute  ↑ attribute value      ↑ content    -->

    <!-- Void element: no closing tag, no children -->
    <img src="photo.jpg" alt="A description" width="400" />

    <!-- Boolean attribute: presence = true, no value needed -->
    <input type="checkbox" checked />

    <!-- Custom data attribute: store data for CSS / JS -->
    <button data-action="save">Save</button>

    <!-- Character entities: use when char has HTML meaning -->
    <p>Use &lt;div&gt; not &lt;span&gt; for block layout.</p>
    <!-- renders: Use <div> not <span> for block layout.   -->

    <!-- Whitespace: multiple spaces / newlines collapse to one -->
    <p>Word    word     word</p>
    <!-- renders: Word word word                              -->

  </body>
</html>

⚠ Common Mistakes

  • Omitting <!DOCTYPE html> triggers quirks mode — browsers render inconsistently
  • Forgetting the closing tag on container elements — browser repairs it, but not always where you expect
  • Using & bare in text or URLs inside attributes — must be &amp;
  • Duplicate id values on a page — ids must be unique; duplicates break CSS, JS, and a11y
  • Omitting lang on <html> — screen readers guess the language, often wrongly

✓ Tips & Best Practices

  • Always start with the HTML5 boilerplate — DOCTYPE, lang, charset, viewport, title
  • Use lowercase for all tag names and attribute names — it's the universal convention
  • Quote all attribute values with double quotes, even when technically optional
  • Use the W3C validator (validator.w3.org) to catch structural errors the browser silently hides
  • Indent nested elements consistently — structure mirrors the DOM tree

Quick Summary

  • HTML = structure & meaning; CSS = appearance; JS = behavior
  • <!DOCTYPE html> must be the very first line
  • Elements are either container (with closing tag) or void (no closing tag)
  • Attributes live in the opening tag: name="value"
  • Browsers silently fix bad HTML — use a validator to surface real errors
