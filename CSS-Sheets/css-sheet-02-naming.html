Naming Conventions

BEM · SMACSS · OOCSS · utility-first · custom properties
Description
CSS naming conventions give classes meaning, prevent conflicts, and make code self-documenting. BEM (Block Element Modifier) is the most widely adopted: it encodes structure directly into class names. SMACSS categorizes rules by role and uses prefixes. OOCSS separates structure from skin. Utility-first (Tailwind-style) uses single-purpose classes. Custom property names follow a consistent --category-variant token pattern.
Notes
BEM uses double-underscore for elements (block__element) and double-hyphen for modifiers (block--modifier)
SMACSS prefixes: l- for layout, is- for state — elements and modules are unprefixed
Utility classes trade HTML readability for zero CSS growth — every class does exactly one thing
Custom properties use --category-variant pattern: e.g. --color-primary, --spacing-md
Example
.card { }                        /* BEM: Block */
.card__title { }                 /* BEM: Element */
.card--featured { }              /* BEM: Modifier */

.l-grid { }                      /* SMACSS: Layout */
.is-active { }                   /* SMACSS: State */

.structure { width: 100%; }      /* OOCSS: structure class */
.skin-dark  { background: #111;} /* OOCSS: skin class */

.mt-4  { margin-top: 1rem; }    /* Utility-first */
.flex  { display: flex; }

:root {
  --color-primary: #5c6bc0;      /* Custom property token */
  --spacing-md:    1rem;
}
Code Explained
.card — BEM block: the standalone, self-contained component
.card__title — BEM element: a part that lives inside the block
.card--featured — BEM modifier: a named variant of the block
.l-grid — SMACSS layout prefix signals structural intent
.is-active — SMACSS state prefix — toggled by JavaScript
.structure / .skin-* — OOCSS separates layout from appearance
.mt-4 — utility class: one rule, one job, no exceptions
--color-primary — token naming: category then variant
BEM Pattern
SyntaxDescription
.blockStandalone component (e.g. .card)
.block__elementChild part (e.g. .card__title)
.block--modifierVariant (e.g. .card--dark)
.block__el--modElement with modifier
SMACSS Prefixes
SyntaxDescription
(none)Base — element defaults, resets
l-Layout — structural containers
(none)Module — reusable UI components
is-State — JS-toggled conditions
theme-Theme — visual skin overrides

SyntaxDescription

--shadow-*e.g. --shadow-card
⚠ Common Mistakes
Mixing conventions in the same project — pick one and commit to it
Using vague names like .box or .thing — names should describe purpose
BEM modifiers that encode visuals: .btn--red — prefer semantic .btn--danger
Skipping the block on elements: .__title instead of .card__title
✓ Tips & Best Practices
BEM for component-based projects; utility-first when paired with Tailwind or similar
Use is- and has- prefixes even in BEM to signal JS-toggled state clearly
Name custom properties by role not value: --color-danger not --color-red
Document your convention in the project README so all contributors stay consistent
Quick Summary
BEM: block__element--modifier — the most widely used convention
SMACSS: categorize by role; prefix layout (l-) and state (is-)
OOCSS: separate structure (layout) from skin (appearance)
Custom properties: --category-variant token pattern throughout
